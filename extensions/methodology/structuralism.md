# 结构主义

## 这是什么

不是"看整体结构"。结构主义的核心洞见是：**很多事物的意义不在它自身，而在它和别的事物的关系网络中。** 孤立地理解一个实体必然导致误解——你必须在关系网络中定位它。文件描述符是一个整数，但它的意义完全来自 `process → fd_table[fd] → file_object → inode → super_block → block_device` 这条关系链。切断关系链，fd 只是一个数字。

> 区分：整体观问"我在系统的哪一层？"，结构主义问"这个东西和哪些东西有关系？这些关系定义了它是什么？"

## 三步操作流程

### Step 1: 找关系——三向展开

对你要理解的目标实体，找出三个方向的关系。

```
[结构主义·Step1]
目标实体：______

上游（谁依赖它/调用它）：
  - ______：通过 ______（接口/方法）依赖它，为了 ______（目的）
  - ______：

下游（它依赖谁/调用谁）：
  - ______：通过 ______（接口/方法）调用，为了 ______（目的）
  - ______：

同层（和它平行的实体）：
  - ______：和它的区别是 ______
  - ______：
```

### Step 2: 做减法——如果去掉它，什么断了？

不是在问"这个功能没了"——是问系统里哪些路径会断掉，断在哪个接口。

```
[结构主义·Step2]
如果去掉 ______：
  - 上游 ______ 的 ______ 路径会断，因为 ______
  - 下游 ______ 不再被使用，但 ______ 不受影响
  - 同层 ______ 需要接管它的职责，但 ______（为什么不行/可行）
这个实体的不可替代性来自：______（哪条关系是不可替代的？）
```

### Step 3: 找意义锚点——这个实体"是什么"由哪条关系决定？

```
[结构主义·Step3]
______ 的"意义"由以下关系决定（按重要性排序）：
1. 和 ______ 的关系决定了它的 ______（核心定义关系）
2. 和 ______ 的关系决定了它的 ______
3. 和 ______ 的关系决定了它的 ______
去掉这些关系，它只是 ______（失去意义的纯粹形态——比如 fd 只是一个 int）。
```

## 四种假执行模式

### 假模式 1：只画框不画边

```
❌ "[结构主义] 系统包含 A、B、C 三个模块。"
   → 列举实体不等于结构主义。"包含"不是关系——它是框。

✅ 每条关系必须标注：方向（谁→谁）、接口（通过什么连接）、目的（为了什么）。
   "A 通过 B.doSomething() 调用 B，目的是获取用户权限——不是 B 的任何其他功能。"
```

### 假模式 2：关系链画了但不做减法

```
❌ 画了完整的关系图，觉得理解了。
   → "看懂了"是假象。做减法才知道你是否真的理解了哪条边是结构性的、哪条是装饰性的。

✅ Step2 的减法测试：想象去掉这个实体——系统哪里断？哪里不断？
   能回答这个问题，才算真的理解了关系网络。
```

### 假模式 3：关系无限扩展——画了 50 个节点

```
❌ "[结构主义] fd 和内存管理有关（因为 file_object 需要 kmalloc），
   和调度有关（因为阻塞 I/O 会睡眠），和网络栈有关（因为 socket 也是 fd）……"
   → 万物互联 = 什么都没说清。

✅ 限定关系范围——只画和当前问题直接相关的关系。
   "当前讨论 fd 的 dup 行为——相关的关系链是：
   process→fd_table→file_object（refcount 在这里）→inode（不随 dup 变化）。
   内存管理、网络栈在此无关。"
```

### 假模式 4：认为关系是静态的

```
❌ 画了一张"系统架构图"觉得这就是系统。
   → 关系在运行时是动态的。fd_table 的映射在 fork 后父子共享 file_object，
   但在 exec 后 close-on-exec 的 fd 会被关闭——关系随操作变化。

✅ 标注关系何时建立、何时解除、何时变化。
   "fork: 子进程复制 fd_table，但 file_object 共享（refcount++）。
   exec: close-on-exec fd 关闭，file_object refcount--。
   close: fd_table[fd] 清空，refcount--，如果归零销毁 file_object。"
```

## 不同领域的示例

### 示例 1: 操作系统 — 文件描述符

```
[结构主义·Step1]
目标实体：文件描述符（fd = 3）

上游：
  - 用户进程：通过 read(3, buf, len) 使用它，目的是读取文件数据
  - 内核：通过 fd_table[current->pid][3] 解析它

下游：
  - file_object：fd_table[3] 指向它，它持有 inode 引用和当前偏移量
  - inode：file_object 通过 inode 访问实际文件数据和元数据

同层：
  - fd=0 (stdin), fd=1 (stdout), fd=2 (stderr)：和 fd=3 的唯一区别是编号和 file_object 指向
  - fd=4：可能是另一个文件或 socket

[结构主义·Step2]
如果去掉 fd：
  - 用户进程无法引用已打开的文件——open() 返回了什么？无法传给 read()
  - file_object 仍然存在但无法被用户态访问
  fd 的不可替代性：它是用户态和内核态之间唯一允许的"文件引用句柄"。
  进程不能直接持有 file_object 指针——那是内核地址空间。

[结构主义·Step3]
fd 的"意义"：
1. 和 fd_table 的关系（整数→指针的映射）——核心定义
2. 和 file_object 的关系（通过指针间接引用）——通过引用计数实现共享
3. 和进程的关系（per-process 的编号空间）——fd 在 fork/exec 时的不同行为来源于此
去掉所有关系，fd 只是一个 int——和 age=42 没有任何区别。
```

### 示例 2: Web — REST API 的 URL

```
[结构主义·Step1]
目标实体：GET /users/123/orders?status=pending

上游：
  - 客户端：通过 HTTP GET 请求该 URL，目的是获取用户 123 的待处理订单
  - 负载均衡器：通过 path 路由到正确的服务

下游：
  - Router：将 /users/:id/orders 映射到 OrderController.listByUser
  - OrderController：通过 UserService 验证用户 123 存在
  - OrderRepository：通过 user_id=123 + status=pending 查询数据库

同层：
  - GET /users/123：查看用户信息（不涉及订单）
  - GET /orders?user_id=123：语义相同但 URL 结构不同（扁平 vs 嵌套）

[结构主义·Step2]
如果去掉 /users/:id/orders：
  - 客户端仍然可以通过 GET /orders?user_id=123&status=pending 获取同样数据
  - 但嵌套 URL 表达了"订单属于用户"的业务层级——这个信息在扁平 URL 中丢失了
  URL 结构的不可替代性在于它编码了资源的归属关系，不只是查询参数。

[结构主义·Step3]
URL 的"意义"：
1. 和 REST 路由表的关系（pattern→handler 映射）——核心定义，决定了哪个代码被调用
2. 和业务模型的关系（orders 属于 user 123）——嵌套结构编码了 owner 关系
3. 和 HTTP 方法的关系（GET 表示读取）——同一个 URL 在不同的 HTTP 方法下含义不同
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 每条关系是否标注了方向、接口、目的？ | 至少上游和下游各有一个具体的关系 | 只有模块名称和箭头，没有接口和目的 |
| 我是否能回答"去掉它，系统哪里断？哪里不断？" | 可以具体说出断在哪条路径上 | "系统会出 bug"——太模糊 |
| 关系的范围是否有边界（不是万物互联）？ | 和当前问题无关的关系被显式排除 | 关系图画了 20+ 个节点 |

## 转换信号

### 何时进入
- 分析一个你不理解的实体——先看它的关系网络
- 设计新模块——先定义它和已有模块的关系
- 用户只关注一个实体的自身属性而忽略了它的关系

### 何时退出
- 三向关系分析完成 + 减法测试通过 → 如果还需要深入理解该实体本身，转 [格物致知] 读源码
- 发现某条关系是关键但未知 → 转 [整体观] 放大到更大的架构图
- 关系分析完成 → 可以做局部深入了

## 标注前自检清单

- [ ] 是否列出了上游（谁依赖我）和下游（我依赖谁）？
- [ ] 每条关系是否标注了接口和目的（不能只是一个箭头）？
- [ ] 是否做了减法测试（去掉它，系统哪条路径断？）？
- [ ] 是否找到了定义这个实体的核心关系（按重要性排序）？

**实体的意义来自关系网络。切断关系，它什么都不是。**
