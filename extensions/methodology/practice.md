# 实践论

## 这是什么

不是"动手试试"。实践论的核心洞见是：**认识不是在脑内完成的——它是在"实践→反馈→修正"的循环中形成的。未经实验验证的结论只是假设，不管推理过程多漂亮。** 实践论的作用是在每一轮分析之后追问一个具体的、可执行的验证动作。

> 区分："你应该加个测试"不是实践论。实践论要求你给出具体的操作步骤——加哪几行代码、跑什么命令、预期看到什么输出、如果输出不同意味着什么。

## 四步操作流程

### Step 1: 判断当前状态 —— 还缺理解还是缺实验？

不是所有问题都需要实验。但如果你已经在理论层面讨论超过一轮而没有外部输入——你缺的不是更多分析，而是一次实验。

```
[实践论·Step1]
当前状态：我们讨论了 ____ 轮，建立了 ____ 个假设/推断。
已经验证过的结论：____（哪些是有实验支撑的？）
未验证的结论：____（哪些还只是推理？）
判断：现在缺的是更多分析还是缺一次实验？→ ______
```

### Step 2: 设计最小验证实验

最小 = 最少代码、最短路径、最确定的结果。好的验证实验是二值的：结果只能是 X 或 Y，中间没有解释空间。

```
[实践论·Step2]
验证实验：
操作：______（具体到：在哪个文件的哪一行加什么代码，或跑什么命令）
预期结果 A：______（如果结果是这样 → 意味着 ______）
预期结果 B：______（如果结果是这样 → 意味着 ______）
耗时估算：______（应该尽可能短——超过 5 分钟的实验不是"最小"）
```

> 关键：实验必须在两种结果下都有信息量。只感兴趣"确认我是对的"的实验不是好实验——真正有价值的实验是能推翻你的假设的。

### Step 3: 执行 + 观察

```
[实践论·Step3]
实际执行结果：______
和预期对比：匹配预期 A / 匹配预期 B / 都不匹配（意外）→ ______
```

### Step 4: 修正理解

根据实验结果更新认知状态。不是"实验结果符合预期所以我对了"——是"实验结果排除了哪些可能性，支持了哪些可能性"。

```
[实践论·Step4]
修正后的理解：
- 升级（被验证）：[推断] ______ → 升级为 [事实]
- 保留（未触及）：______ → 仍为 [推断] 或 [假设待验证]
- 排除（被推翻）：______ → 排除
- 新问题（意外结果）：______ → 需要新一轮溯因
```

## 四种假执行模式

### 假模式 1：设计不可证伪的实验

```
❌ "[实践论] 我们可以跑一下 busybox 看看。
   如果 page fault 还在，说明 bug 没修好。"
   → 跑一下"看看"不是实验。"bug 还在"这个结果不能告诉你新信息。

✅ "验证实验：在 bget 复用路径加 b->valid = 0，重新编译内核，
   启动 busybox。如果 page fault 消失 → 确认该路径是不变量破坏点。
   如果 page fault 仍然存在 → valid 不是唯一问题（还有其他破坏点）。"
   两种结果都有明确的信息量。
```

### 假模式 2：实验太大，反馈太慢

```
❌ "[实践论] 我们可以写一个完整的单元测试覆盖所有 block cache 操作。"
   → 在你写完测试之前，问题可能已经通过更简单的 probe 被定位了。

✅ 最小实验：一行 printk 在 bget 复用路径，打印"复用 buffer: old=(%d,%d) new=(%d,%d) valid=%d"。
   重新编译，启动 busybox，看日志。5 分钟拿到结论。
   复杂测试是验证修复时用的，不是定位根因时用的。
```

### 假模式 3：结果不符合预期时自圆其说

```
❌ "实验结果不符合预期……可能是实验环境有问题。"
   → 用"可能"回避了实验推翻了假设的事实。

✅ 实验推翻了假设 = 进展。显式宣布："这个实验排除了假设 A。
   可能性空间缩小了。接下来验证 B。"不自圆其说，修改假设。
```

### 假模式 4：分析太多，实验太少

```
❌ 连续 5 轮对话全在分析，没有一次提到"我们加个 probe 跑一下"。
   → 纯脑内推理是收敛的——越来越精细也越来越脱离实际。

✅ [实践论] 的触发阈值很低：在同一个问题上分析超过 2 轮还没做实验，
   必须插入验证。"分析暂停——我们加个 printk 跑一下，
   比再猜 2 轮有效。"
```

## 不同领域的示例

### 示例 1: 后端 — 怀疑数据库连接泄漏

```
[实践论·Step1]
讨论了 2 轮：① 连接池配置看起来正常 ② 怀疑某个代码路径没释放连接
未验证：实际运行时连接池的状态
判断：缺实验。讨论再多也看不到哪个路径泄漏了连接。

[实践论·Step2]
验证实验：
操作：在连接池的 get() 和 release() 各加一行日志，
     打印 ThreadName + 调用栈的前 3 帧 + 当前活跃连接数。
     部署到测试环境，跑一轮压测。
预期 A：日志显示某个 Thread 只 get 不 release → 泄漏路径找到了
预期 B：所有 get 都有对应的 release → 泄漏不在应用层，在连接池本身或网络层
耗时：加日志 5 分钟 + 部署 10 分钟 + 压测 1 分钟

[实践论·Step3]
执行结果：Thread-http-12 在 UserService.updateEmail 路径上 get 了连接，
没有对应的 release。该路径在 sendEmail 失败时抛出了异常但没进 finally 块。

[实践论·Step4]
- 升级：[推断] 连接泄漏 → [事实] Thread-http-12 路径存在泄漏
- 排除：连接池配置问题（配置正确，个别路径泄漏）
- 修复：sendEmail 异常路径加上 try-finally 确保 release
```

### 示例 2: 前端 — 怀疑组件不更新的原因是 memo

```
[实践论·Step2]
验证实验：
操作：在 DashboardTable 组件中临时移除 React.memo 包裹（一行改动），
     刷新页面，看 props.data 更新后 UI 是否跟着变。
预期 A：移除 memo 后 UI 更新了 → memo 的比较函数有问题
预期 B：移除 memo 后 UI 还是不变 → 问题在父组件的 props 传递，不是 memo
耗时：改一行代码 + 刷新浏览器 = 30 秒
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 我的实验在两种结果下都有信息量吗？ | 写出"结果 A→意味着 X"和"结果 B→意味着 Y" | 只写了"如果结果符合预期就验证了" |
| 实验是最小的吗？ | 耗时 < 5 分钟，改动 < 5 行代码 | "写一个完整的测试套件"或"重构之后再测" |
| 实验结果有没有更新认知状态？ | 每条相关 [推断] 被升级/排除/保留 | 实验做了但理解没变 |

## 转换信号

### 何时进入
- 任何分析/假设形成后 → 必须追问"最小验证实验是什么"
- 连续 2 轮以上纯理论讨论 → 强制插入实践论
- 用户长时间在"思考"但没有外部反馈 → 建议动手

### 何时退出
- Step4 完成认知状态更新 → 如果结论已锁定，转 [收束] 或 [辩证法] 回看
- 实验结果意外 → 转 [溯因] 用新现象重新形成假设
- 实验推翻了当前假设 → 转 [收束] 排除一个方向 + [溯因] 新假设

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [溯因] 形成假设后 → | **实践论** 验证 | [溯因] 实验结果意外 → 新溯因 |
| [收束] 确定验证方向后 → | | [收束] 实验排除一个方向 → 宣布收束 |
| [第一性原理] 推导出最小方案后 → | | [辩证法] 被推翻后的回看 |

## 标注前自检清单

- [ ] 是否给出了具体的操作步骤（文件、行、命令）而非"你应该测试一下"？
- [ ] 实验是否在两种结果下都有明确的信息量？
- [ ] 这是当前最小可做的实验吗？（能不能更简单？更快？）
- [ ] 实验结果有没有导致认知状态的更新（升级/排除/产生新问题）？
- [ ] 如果实验推翻了假设，我是宣布"排除"了还是在自圆其说？

**"分析暂停，先加个 probe 跑一下"——这是实践论最有力的介入。**
