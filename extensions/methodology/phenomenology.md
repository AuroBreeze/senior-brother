# 现象学观察

## 这是什么

不是"看看发生了什么"。现象学观察是一个反直觉的纪律：**在收集到足够事实之前，禁止解释。** 人类的默认模式是看到一个现象立刻跳到解释——这个模式在 debug 中是致命的，因为错误的解释会过滤掉你不愿意看到的事实。

> 核心洞察：Debug 慢的人不是不会推理，是观察太粗糙。他们看到一个现象，立刻有了一个解释，然后只收集支持这个解释的证据。现象学观察强制你在解释之前先铺开全部事实——不管这些事实和你的直觉是否冲突。

## 四步操作流程

### Step 1: 划观察边界

先说清楚你在看什么范围。不是"整个系统"，而是具体的时间窗口、具体的组件、具体的输入条件。

```
[现象学·Step1] 观察边界：
- 时间：______ 到 ______
- 组件：______（具体模块/进程/函数）
- 条件：______（什么输入？什么配置？什么环境？）
- 我明确不在此次观察范围内的是：______
```

> 边界划得越小，观察越精确。"整个系统"的观察 = 没有观察。

### Step 2: 列出纯事实（不解释！）

只列可以通过观察/测量/日志/寄存器直接确认的内容。每条事实标注来源。

```
[现象学·Step2] [事实] 清单：
1. [事实] ______（来源：______）
2. [事实] ______（来源：______）
3. [事实] ______（来源：______）
```

> 问自己的过滤问题："如果我把这条给别人看，他会质疑这是事实还是我的理解？" 如果有人会质疑——这不是 [事实]，这是 [推断]。

### Step 3: 列出推断（与事实分开！）

基于事实你有了什么理解？标 [推断] 并注明它基于 Step2 中的哪条事实。

```
[现象学·Step3] [推断] 清单：
1. [推断] ______（基于事实 #______）
2. [推断] ______（基于事实 #______，需验证）
```

### Step 4: 标注空缺

明确说出"我还不知道什么"。这张清单比已知清单更重要——它决定了下一步往哪走。

```
[现象学·Step4] 当前未知：
- ______（如果知道了，会显著改变推断方向）
- ______（次要，但有助于确认）
```

## 四种假执行模式

### 假模式 1：把推断当成事实列

```
❌ "[事实] 缓存返回了错误数据"
   → 这是推断。你看到的现象是数据不对，缓存是推断的原因。

✅ "[事实] 用户内存中 0x1049fc 处的内容是 0xe101b783"
✅ "[事实] 磁盘文件 0x1049fc 处的内容是 0x00d71463"
✅ "[推断] 用户内存内容 ≠ 磁盘文件内容（基于以上两条事实）"
✅ "[推断] 差异可能源于缓存层（基于调用链分析，待验证）"
```

### 假模式 2：现象没列完就开始解释

```
❌ "epc 处是 addi 但还是 page fault 了，肯定是内存不对。"
   → 还有一个关键现象没列：tval=0x128。0x128 不是一个随机值，
   它是 296 的十进制——这个数字在后面会成为关键线索。
   如果跳过了这个"细节"，后面的溯因会走弯路。

✅ 把寄存器 dump 的每一个字段都当成可能重要的现象。
   现在不知道什么是重要的——所以全部列出来。
```

### 假模式 3：只列"相关"现象——但"相关"是你判断的

```
❌ "我只列了和 page fault 相关的寄存器。"
   → "相关"是你基于一个未意识到的假设判断的。你可能过滤掉了反例。

✅ 多列 20 条无关事实的代价远低于漏掉 1 条关键事实。
```

### 假模式 4：跳过"未知清单"

```
❌ 事实列完了，推断也列了，直接开始溯因。
   → 未知清单是你的认知边界标记。不画边界就进入溯因 = 在不知道
   自己不知道什么的情况下做推理。

✅ 未知清单至少 3 项。如果列不出 3 个"还不知道的事"，你不是都知道了——
   你是不知道自己不知道什么。
```

## 不同领域的示例

### 示例 1: 网络调试 — TCP 连接莫名断开

```
[现象学·Step1] 观察边界：
- 时间：2026-05-24 14:30-14:35 UTC
- 组件：client(10.0.1.5:45678) ↔ server(10.0.2.1:443)
- 条件：TLS 1.3，已建立连接 120 秒，空闲 30 秒后断开

[现象学·Step2] [事实] 清单：
1. [事实] client 端 tcpdump 显示收到了 RST（来源：pcap 文件，包 #1047）
2. [事实] RST 的 TTL=64，和之前 server 发的包 TTL 一致（来源：同一 pcap）
3. [事实] server 端应用日志没有异常（来源：server app.log）
4. [事实] client 端 socket 错误是 "Connection reset by peer"（来源：client error log）
5. [事实] 连接在空闲 30 秒后断开，不是精确 30 秒（来源：pcap 时间戳，29.8s-31.2s 范围）

[现象学·Step3] [推断] 清单：
1. [推断] RST 确实来自 server（基于 #1, #2——TTL 一致说明不是中间设备注入的）
2. [推断] server 应用层没有主动关闭（基于 #3）
3. [推断] 空闲超时可能来自 server 端的某个中间层（基于 #5——不是精确 30 秒，不像定时器）

[现象学·Step4] 当前未知：
- server 端是否有负载均衡器/代理在中间？（如果知道了会显著改变方向）
- server OS 的 tcp_keepalive 配置是什么？
- 为什么是 30 秒而不是更常见的 60/300 秒？
```

### 示例 2: 前端 — 组件数据不更新

```
[现象学·Step1] 观察边界：
- 组件：DashboardTable
- 条件：props.data 更新后，UI 没有反映新数据

[现象学·Step2] [事实] 清单：
1. [事实] console.log(props.data) 在父组件中输出的是新数组（长度 15）
2. [事实] DashboardTable 中的 console.log(props.data) 输出的是旧数组（长度 10）
3. [事实] React DevTools 显示 DashboardTable 的 props.data 是旧数据
4. [事实] 父组件的 useEffect 依赖数组包含了 data

[现象学·Step3] [推断] 清单：
1. [推断] 父组件的状态确实更新了（基于 #1）
2. [推断] props 在传递过程中没有被正确更新（基于 #2, #3）
3. [推断] 可能是 memo/useMemo/useCallback 某处阻止了重渲染（基于 #2）

[现象学·Step4] 当前未知：
- DashboardTable 是否被 React.memo 包裹？
- 父组件传给 DashboardTable 的 props 除了 data 还有什么？
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 我能在事实和推断之间画一条清晰的线吗？ | 每条陈述都能区分 | "事实"里有"因为""所以""应该是"等解释性语言 |
| 我是否列了至少 3 个"还不知道"？ | 3+ 项 | 未知清单为空或只有 1 项"根因是什么" |
| 我是否看到了和我直觉不一致的现象但不打算忽略它？ | 至少检查过一遍有没有反直觉现象 | 现象清单全部指向同一个方向——你可能过滤了 |

## 转换信号

### 何时进入
- 开始 debug 任何问题 → 第一步永远是现象学
- 对话中用户在解释和猜测之间跳跃 → 拉回来："先停一下，把事实列出来"
- 排查方向卡住 → 回到现象学，可能有遗漏的事实
- 用户说了很多信息但没有结构化 → 帮他整理事实/推断/未知

### 何时退出
- 事实清单已经支撑得起至少 2 个互斥假设 → 转 [溯因推理]
- 发现事实不够，需要更多观测 → 转 [实践论] 设计 probe/实验
- 发现某个"事实"其实是推断，依赖未验证的前提 → 设计验证实验确认

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [整体观] 先知道自己在看哪一层——不知道观察边界，现象收集没有焦点 | **现象学观察** | [溯因推理] 事实够了就形成假设 |
| | | [还原论] 如果现象太多，拆成子系统分别观察 |
| | | [实践论] 如果事实不够，设计实验获取更多事实 |

## 标注前自检清单

- [ ] 我是否把 [事实] 和 [推断] 分开了（物理上分开，不是混在一段里）？
- [ ] 是否有任何 [事实] 其实是推断但仍然标了 [事实]？
- [ ] 未知清单是否有至少 3 项具体的"还不知道"？
- [ ] 我是否在收集事实的过程中忍住了不去解释？
- [ ] 如果有一条现象和我的预期矛盾，我是否把它列出来了而不是忽略？

**任何一条未通过，不要进入解释阶段。继续观察。**
