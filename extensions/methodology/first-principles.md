# 第一性原理

## 这是什么

不是"回归基本原理"的口号。第一性原理是一个具体的思维操作：**把你正在分析的东西先拆掉，看如果没有它，什么约束会导致它必须以某种形式重新出现。** 推导的起点必须是不可再退的约束——物理定律、计算边界、接口契约——不能是另一个同层概念。

> 区分：从"虚拟内存需要页表"出发 = 循环论证（虚拟内存这个概念本身已经假定了地址翻译）。从"程序想要连续的地址空间，物理内存是碎片化的，CPU 每条访存指令都需要一个物理地址"出发 = 第一性原理。前者的起点是概念，后者的起点是约束。

## 五步操作流程

每一步都是具体动作，不是思维姿态。执行时在对话中显式标注步骤编号。

### Step 1: 指名要拆解的对象

说出你正在分析的具体设计/机制/规则。不能是泛化的类别（"虚拟内存"），必须是具体的（"为什么 RISC-V Sv39 页表是三级而不是两级？"）。

```
[第一性原理·Step1] 我要拆解的是：______（一个具体的设计决策）
```

### Step 2: 假设它不存在——精确描述什么会坏掉

不可以说"系统会崩溃"。必须精确到：哪个操作会失败？失败在哪个环节？为什么其他机制无法代偿？

```
[第一性原理·Step2] 如果没有 ______，在 ______ 这个操作中，______ 会因为 ______ 而失败。这不是 ______ 能解决的问题，因为 ______。
```

模板中五个空依次是：机制名、具体操作、哪个参与方、什么原因、替代机制、替代机制为什么不行。

> 如果你填不满这五个空，说明你还不够理解这个机制解决的问题。退回去重新观察。

### Step 3: 列出不可绕过的约束

分三类列。每条约束必须是可以独立验证的陈述，不能依赖你要解释的机制本身。

| 约束类型 | 约束陈述 |
|---------|---------|
| 物理约束 | ______（如：CPU 只能访问物理地址） |
| 计算约束 | ______（如：遍历 2^27 个页表项不可能每次查一遍） |
| 接口约束 | ______（如：应用通过 syscall 申请内存，不直接管理物理页） |

### Step 4: 从约束推导最小可行方案

不引入任何不服务于上述约束的复杂性和不引入任何不服务于上述约束的复杂性。推导出来的方案可能和实际实现不同——这是正常的关键步骤，差距本身就是信息。

```
[第一性原理·Step4] 给定以上约束，最小方案是：______。
它的核心机制是：______（一句话）。
它不包含：______（列举当前实际实现中有但最小方案中不需要的部分）。
```

### Step 5: 对比实际实现——解释差距

实际实现比最小方案多了什么？多出来的每一部分，原因是什么（历史遗留、性能优化、工程便利、安全要求、向后兼容）？

```
[第一性原理·Step5] 实际实现比最小方案多了：______。
这些多出来的部分来源于：______（约束列表中不包含，但实际工程中存在的附加约束）。
```

## 四种假执行模式

以下四种是 AI 最容易滑入的模式。每种都**标注了 `[第一性原理]` 但实际上没有执行推导**。

### 假模式 1：概念循环

```
❌ "[第一性原理] 文件系统需要 inode 来索引文件数据。"
   → inode 是文件系统的一部分，用 inode 解释文件系统 = 循环。

✅ "[第一性原理·Step2] 如果没有一种方式把文件名映射到磁盘块号，
   open("/foo/bar") 时 OS 不知道读哪个扇区。文件名的字符串→磁盘块号
   的映射是必须存在的，inode 是这种映射的一种实现。"
```

### 假模式 2：诉诸惯例

```
❌ "[第一性原理] 因为业界标准做法就是三级页表。"
   → "标准做法"不是约束，是其他人在其他约束下的选择。

✅ "[第一性原理·Step3] 物理约束：CPU 只能访问物理地址。
   接口约束：应用看到的是连续虚拟地址。
   计算约束：存储 2^27 个页表项的直接映射表需要 512MB（不可接受）。
   所以必须有一个稀疏的、分级的查找结构。"
```

### 假模式 3：跳过推导直接跳到结论

```
❌ "[第一性原理] 所以 RISC-V 选择 Sv39 是合理的。"
   → 没有推导过程，只是给结论贴了标签。

✅ 必须展示 Step2→Step3→Step4 的完整链路。中间有不确定的地方
   标注 [推断] 或 [假设待验证]，不要用 [第一性原理] 掩盖推理缺口。
```

### 假模式 4：把性能优化当成第一性约束

```
❌ "[第一性原理] 这个结构体必须 64 字节对齐因为缓存行是 64 字节。"
   → 缓存行大小不是第一性约束——有没有缓存都可以工作，只是慢。

✅ "缓存行对齐是性能优化，不是第一性约束。真正的第一性约束是：
   CPU 按字访问内存，不对齐访问在 RISC-V 上会触发异常。"
```

> 判断标准：去掉这个约束，系统还能不能工作（即使慢、即使丑）？如果能——它不是第一性约束，是工程约束。

## 不同领域的示例

### 示例 1: OS 内核 — 为什么内核需要独立的栈？

```
[第一性原理·Step1] 要拆解：内核为什么不用用户程序的栈？

[第一性原理·Step2] 如果内核和用户共享同一个栈——
在系统调用陷入内核时，SP 指向用户栈顶。
用户可以把 SP 设为一个非法地址（如 0x0），
然后触发 syscall。内核第一件事是 push 寄存器到栈上——
push 到 0x0 → 内核崩溃。
这不是"内核小心一点"能解决的——内核必须在知道 SP 是否合法之前
就开始保存状态。

[第一性原理·Step3]
- 物理约束：CPU 只有一个 SP 寄存器，push/pop 指令隐式依赖 SP
- 接口约束：用户态可以任意修改 SP（这是用户态的权限）
- 计算约束：内核无法在 push 之前验证用户 SP 的有效性（验证本身需要寄存器）

[第一性原理·Step4] 最小方案：陷入内核时，硬件自动切换到另一个 SP
（由内核控制，用户不可写）。不需要两个栈全部隔离——只需要陷入的
第一条指令使用内核 SP。

[第一性原理·Step5] 实际实现（完全独立的 kernel stack per CPU）
比最小方案多了：per-CPU 而非共享（并发约束），
完整栈而非仅入口（内核内部函数调用需要完整栈帧）。
```

### 示例 2: 数据库 — 为什么需要 WAL？

```
[第一性原理·Step1] 要拆解：为什么数据库不直接写数据页而要先写日志？

[第一性原理·Step2] 如果没有 WAL，每次事务提交时必须把被修改的
所有数据页（可能散落在磁盘各处）完整写回。一个事务改了 10 个页 =
10 次随机写。如果在这 10 次写的中途崩溃——数据页一半是新的、
一半是旧的，没有恢复手段。undo log 也不行，因为旧值在崩溃前
可能已经被覆盖。

[第一性原理·Step3]
- 物理约束：磁盘的原子写单位是扇区（512B/4KB），不是事务
- 计算约束：无法在一次 fsync 中原子地更新多个不连续的数据页
- 接口约束：事务提交的语义是"要么全做，要么全不做"

[第一性原理·Step4] 最小方案：在写数据页之前，先把"我要改什么、
改成什么样"写到磁盘上一个连续区域。崩溃后读这个区域 = 知道哪些
事务完成了，哪些需要 redo。

[第一性原理·Step5] 实际实现多了：LSN、checkpoint、WAL 归档、
逻辑解码——这些是性能、恢复速度、复制等工程约束加上去的。
最小 WAL 只需要 append-only 日志 + redo 扫描。
```

### 示例 3: 前端 — 为什么 React 需要虚拟 DOM？

```
[第一性原理·Step1] 要拆解：为什么 React 不直接操作真实 DOM？

[第一性原理·Step2] 如果没有虚拟 DOM，状态变化时你只有两个选择：
① 精确找到哪个 DOM 节点需要更新（需要手动维护状态→DOM 的映射，
   应用复杂时这本身就是 bug 来源）；
② 每次状态变化替换整个子树（innerHTML）——简单但慢（重新解析 HTML、
   重建节点、丢失内部状态如焦点和滚动位置）。

[第一性原理·Step3]
- 计算约束：全量替换 DOM 的开销（HTML 解析 + 样式重算 + 布局）
  在每次交互时都是不可接受的
- 接口约束：真实 DOM 的状态不仅包含属性，还包含运行时状态
  （焦点、选区、滚动位置、iframe 内容）
- 物理约束：浏览器提供的 DOM API 是命令式的，没有声明式接口

[第一性原理·Step4] 最小方案：在内存中保留一棵"上一次渲染的树"，
状态变化时比较新旧树的差异，只把差异应用到真实 DOM。
树是什么格式不重要——JS 对象即可。

[第一性原理·Step5] 实际实现的 React 比最小方案多了：
Fiber（调度）、hooks（组合）、并发模式（中断渲染）。
这些来源于：帧率要求（16ms）、复杂应用的优先级管理、
可组合性优于继承的工程约束。
```

## 深度自检：你是在推还是只是在说？

在标注 `[第一性原理]` 之前，用以下三个问题自测：

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 我是否说出一条约束，它被违反后系统**完全不能工作**（而非"会变慢"）？ | 至少一条硬约束 | 约束全是"性能""业界标准""可维护性" |
| 约束是否可以在不提及当前机制的情况下陈述？ | 每条都可以 | 约束里出现了你要解释的机制的名字 |
| 从我的约束出发，是否可能推出不同的设计？ | 应该是 | 如果只有一种推导结果，你可能是先知道结论再倒推的 |

> 三条有一条不及格——你做的不是第一性原理，是事后合理化。不要标注 `[第一性原理]`，改标 `[推断]`。

## 转换信号

### 何时进入

- 用户问"为什么需要 X？"——直接进入 Step1
- 你在解释一个设计但没有追溯到约束层——停下来，进入 Step2
- 对话在概念间打转（"A 用 B，B 依赖 C"但没有说为什么）——切换到第一性原理
- 比较两个方案优劣时——对每个方案走一遍五步

### 何时退出

- Step4 的推导完成，且 Step5 的差距已解释 → 如果需要验证结论，转 [实践论]
- 发现自己列不出 Step3 的硬约束（全是软约束）→ 你对这个系统的理解还不够，转 [现象学观察] 或 [格物致知] 去读源码
- Step5 发现差距来自历史路径 → 转 [历史方法] 追溯演化过程
- 推导出的是一个"最合理"而非"唯一可能"的结论 → 转 [溯因推理] 形成假设并验证

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [整体观] 先定位全局位置——不知道自己在分析系统的哪一层，第一性原理找不到约束边界 | **第一性原理** | [实践论] 推导出的最小方案是否正确？加个实验验证 |
| [现象学观察] 先收集事实——不了解实际行为就推导，容易推到错误的约束 | | [历史方法] 如果 Step5 差距很大，去看演化过程 |
| [还原论] 先拆解——太大的系统约束太多，拆成子问题逐一分析 | | [溯因推理] 如果推导不唯一，形成假设并验证 |

> 不要和 [类比锚定] 混淆：[类比] 是用日常直觉解释概念，[第一性原理] 是从约束重新推导。类比帮助理解，第一性原理帮助验证。

## 标注前自检清单

在对话中使用 `[第一性原理]` 标注之前，逐条确认：

- [ ] 我是否执行了五步中的至少前三步？（可以不展示全部，但不能跳过推导）
- [ ] 我列出的约束中，至少有一条是"违反后系统完全不能工作"的硬约束？
- [ ] 我的约束陈述中，有没有出现我要解释的机制的名字？（有 = 循环论证，重写）
- [ ] 我是否区分了硬约束和工程约束（性能、惯例、兼容）？
- [ ] 如果别人从同样的约束推导出不同的设计，我能否说出我的推导链在哪里分叉？

**任何一条未通过，不要标注 `[第一性原理]`。改标 `[推断]` 或 `[猜测]`，然后继续推进理解。**
