# 奥卡姆剃刀

## 这是什么

不是"简单的解释更可能是对的"。奥卡姆剃刀是一个**验证顺序决策工具**：当你面对多个互斥假设时，先验证需要额外假设最少的那个——不是因为它在客观上更可能正确，而是因为它的验证成本更低（需要的假设少 = 验证这些假设本身也是成本），如果排除了它可以最快地缩小可能性空间。

> 关键区分：奥卡姆剃刀不是真理判决——它不告诉你哪个假设是对的。它告诉你先查哪个。如果最简单的假设被排除了，立刻转向下一个。

## 三步操作流程

### Step 1: 列出所有假设 + 每个需要的额外假设

额外假设 = "除了我们已经确认的事实之外，这个解释成立还需要什么额外的条件为真？"

```
[奥卡姆·Step1]
假设 A：______
  额外假设：
    1. ______（可独立验证？是/否）
    2. ______（可独立验证？是/否）
  总假设数：____，可独立验证的：____
假设 B：______
  额外假设：
    1. ______
  总假设数：____，可独立验证的：____
假设 C：______
  ...
```

### Step 2: 按"最便宜先验证"排序

验证成本 = 验证假设所需的操作成本 + 额外假设的验证成本（如果额外假设本身也需要验证）。

```
[奥卡姆·Step2]
验证顺序：
1. 先验证假设 ____：验证成本最低（操作：______，耗时：______）
   如果排除了 → 空间从 {____} 缩小到 {____}
2. 再验证假设 ____：成本中等
3. 最后验证假设 ____：成本最高，但前面的排除让它成为唯一剩的
```

### Step 3: 验证 + 收束

```
[奥卡姆·Step3]
验证结果：
假设 ____：验证 ____ → ✓（保留）/ ✗（排除）
  如果排除 → 不需要再管假设数更少的了 → 下一个。
  如果成立 → 不急着宣布胜利，先看有没有假设数更少的也被保留了？
    没有 → 可以锁定。
```

## 四种假执行模式

### 假模式 1：把"简单"等同于"我熟悉"

```
❌ "[奥卡姆] 我觉得最简单的原因是空指针——因为我经常遇到。"
   → "我熟悉"不等于"假设少"。空指针也需要假设"某个指针没被初始化"。

✅ 数假设，不凭感觉。"假设 A（空指针）需要 2 个额外假设：
   ① 某处指针未初始化 ② 该指针在 page fault 之前被解引用。
   假设 B（缓存返回错误数据）需要 1 个额外假设：
   ① 缓存复用时状态未清零。假设 B 更少——先验证 B。"
```

### 假模式 2：拒绝考虑多假设的解释

```
❌ "[奥卡姆] 需要一个假设以上的解释都太复杂了，不可能是对的。"
   → 自然界和工程系统中，很多正确解释都需要多个前提条件。
   奥卡姆不判定对错，只决定验证顺序。

✅ "假设 A 需要 3 个假设，假设 B 需要 1 个。如果 B 排除了——
   不要因为 A 需要 3 个假设就拒绝验证它。3 个假设中可能有 2 个是已知事实。"
```

### 假模式 3：最简单的假设被排除了还坚持

```
❌ "验证了假设 B（最简单的），结果不符合预期……可能是我测错了。"
   → 最简单的假设被数据推翻了——接受它，转向下一个。

✅ "假设 B 排除了。可能性空间缩小。下一个最简假设是 A（2 个假设）。
   验证 A 的方式是 ______。"
```

### 假模式 4：混淆"额外假设数量"和"描述长度"

```
❌ "[奥卡姆] 假设 A 能用一句话说清，假设 B 需要一段话。所以 A 更简单。"
   → 描述长度和额外假设数量不是一回事。"上帝创造了世界"只有一句话，
   但它需要的额外假设（上帝存在）极其巨大。

✅ 只数"这个解释成立还需要多少不包含在已有事实中的条件"。
   不是数字数，是数未被验证的前提。
```

## 不同领域的示例

### 示例 1: 后端 — API 返回 500

```
[奥卡姆·Step1]
已知事实：API /users/id 返回 500，错误日志显示 NullPointerException，
user 表中有 id=12345 的记录。

假设 A（最近部署引入的 bug）：
  额外假设：① 最近的部署修改了相关代码路径
  ② 修改后的代码在 id=12345 这个特定数据上有 NPE
  总假设数：2

假设 B（数据库连接偶尔失败导致返回 null）：
  额外假设：① 连接池有问题 ② 失败只在特定时间发生
  ③ 返回 null 后被上层代码解引用
  总假设数：3

假设 C（数据本身有问题——user 的某个字段为 null）：
  额外假设：① user 记录的某个非空字段实际上为 null
  总假设数：1

[奥卡姆·Step2]
先验证 C：SELECT * FROM users WHERE id=12345 —— 一行 SQL，10 秒。
如果 C 排除 → 再查 A（git log + 代码审查，5 分钟）。
最后查 B（连接池监控，需要部署监控脚本）。

[奥卡姆·Step3]
验证 C：user 记录中 email 字段为 NULL，但代码中 user.email.toLowerCase()
未做空检查 → NPE。假设 C 成立，无需继续 A 和 B。
```

### 示例 2: 前端 — 用户反馈"页面空白"

```
[奥卡姆·Step1]
已知事实：页面空白，DevTools Console 无报错，Network 面板显示所有资源 200。

假设 A（JS 运行时错误被 try-catch 吞了）：
  额外假设：① 某处有未正确处理的异常 ② 异常被捕获但没记录日志
  总假设数：2

假设 B（React 树根节点条件渲染为 null）：
  额外假设：① 某个顶层条件（如 user==null）被触发
  总假设数：1

假设 C（CSS 导致所有元素不可见——opacity:0 或 display:none 被应用到根节点）：
  额外假设：① 有全局样式覆盖了可见性
  总假设数：1

[奥卡姆·Step2]
B 和 C 各 1 个假设，并列最简。先验证 B（React DevTools 看组件树，30 秒），
再验证 C（Elements 面板检查 body 下的元素样式，30 秒）。

[奥卡姆·Step3]
验证 B：React DevTools 显示组件树正常渲染 → B 排除。
验证 C：body 下所有元素 style 正常 → C 排除。
只剩 A——在可疑组件加 console.log 定位被吞的异常。
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 每个假设的额外假设是否都列出来了？ | 每个假设下都有编号列表 | 只写了假设本身，没数假设数 |
| 验证顺序是否按"假设数少 + 验证成本低"排列？ | 排出了明确的先后顺序 | "这几个都有可能"——没排序 |
| 最简单的假设被排除后，是否转向了下一个？ | 不坚持，不自我怀疑 | "可能是我验证方式不对"——在没有理由质疑验证时 |

## 转换信号

### 何时进入
- 面对 2+ 个互斥假设，不知道先验证哪个
- 讨论发散了，出现了太多可能性
- 有人说"可能是 X 也可能是 Y 还可能是 Z"——但没有优先级

### 何时退出
- 排出了验证顺序 → 转 [实践论] 按顺序执行验证
- 所有假设都被排除 → 回到 [收束] 重新划可能性空间（可能漏了方向）
- 一个假设验证成立且无更简假设成立 → 转 [收束] 锁定

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [溯因] 生成了多个假设 | **奥卡姆剃刀** 排优先级 | [实践论] 按优先级验证 |
| [收束] 列出了可能性空间 | | [收束] 验证排除后宣布缩小 |

## 标注前自检清单

- [ ] 每个假设的额外假设是否都显式列出来了（不是凭感觉"这个更简单"）？
- [ ] 额外假设的数量是不是数的"不包含在已有事实中的条件"？
- [ ] 验证顺序是否按"额外假设少 + 可直接验证"排列？
- [ ] 如果最简单的假设被排除了——我是否明确说了"下一个"？

**奥卡姆剃刀是验证策略，不是真理标准。先查最便宜的，不是只查最便宜的。**
