# 奥卡姆剃刀

## 触发条件

- 面对多个可能的解释/假设
- 有人在用复杂理论解释一个简单现象
- 设计讨论中方案越来越复杂
- Debug 时出现了太多嫌疑方向

## 关键问句

- 哪个解释需要的额外假设最少？
- 哪个解释最容易验证？
- 我是不是在用复杂解释逃避简单错误？（"CPU 有问题" vs "我指针传错了"）
- 如果最简单的解释成立，我们还需要引入复杂解释吗？

## 使用方式

以 `[奥卡姆]` 标注。列出所有假设及其需要的额外假设数量，先验证假设最少的那个。

## 示例

> `[奥卡姆]` 现在有三个可能解释：① CPU 行为异常，② 编译器生成了奇怪代码，③ 自己的指针传错了。需要的额外假设：① 需要假设 CPU 有 bug（极低概率），② 需要假设编译器在 `-O0` 下优化出错（罕见），③ 只需要假设自己的代码对 SP 指向的假设不正确（常见）。先查 ③。不是因为 CPU/编译器绝对没问题，而是因为"自己的代码错了"这个假设最便宜，需要的额外假设最少，也最容易验证。

> ⚠️ 奥卡姆剃刀不是"最简单的解释一定正确"——它只是指导验证顺序。先验证最便宜的假设。
