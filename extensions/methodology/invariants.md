# 不变量思维

## 这是什么

不变量是一条在任何操作前后都必须为真的断言。**绝大部分 bug 的本质都是某条不变量被违反了，但违反的时刻和后果暴露的时刻不在同一处。** 不变量思维就是在代码中显式找出这些断言，然后问：谁负责维护它？它在哪个环节被破坏了？

> 区分：不变量不是"代码规范"或"最佳实践"。不变量是"如果这条不为真，系统在某处一定会出错——也许不是现在，也许不是每次，但一定会"。它比 bug 深一层：bug 是不变量被破坏的症状，不变量是 bug 的根结构。

## 四步操作流程

### Step 1: 找出不变量

面对有问题的代码，问：这段代码依赖什么前提？如果某个前提被违反，会有什么后果？

```
[不变量·Step1] 不变量陈述：
"在 ______（具体位置/状态），______（具体条件）必须为真。
 如果这条不为真，______（具体后果，不是'出 bug'，而是哪个操作会得到什么错误结果）。"

不变量发现位置：______（哪个函数/模块/接口维护它）
```

### Step 2: 找维护者

每条不变量都有一个（或几个）维护者。谁负责保证这条不变量成立？

```
[不变量·Step2]
维护者：______（具体函数/模块/流程）
维护时机：______（在什么时候确保不变量成立？初始化？每次操作后？）
```

### Step 3: 找所有破坏点

哪些操作可能破坏这条不变量？不只看出 bug 的那一处——列出所有可能破坏它的位置。

```
[不变量·Step3] 潜在破坏点：
1. ______（当前怀疑的破坏点 ← 如果是 bug 排查场景，标出当前线索指向哪里）
2. ______（虽然这次没问题，但逻辑上有风险）
3. ______（当前可能不会触发，但条件变化后可能触发）
```

### Step 4: 断言化

能不能在代码中加一个 assert/检查，让不变量被破坏时立刻暴露而不是延迟到后果显现？

```
[不变量·Step4] 可加的检查：
位置：______（具体文件:行号附近）
检查内容：______（具体条件表达式）
如果这行检查存在，问题会在 ______（破坏点）暴露，而不是在 ______（后果暴露点）暴露。
```

## 四种假执行模式

### 假模式 1：把"建议"当不变量

```
❌ "[不变量] 函数应该检查返回值。"
   → "应该"不是不变量。不检查返回值在某些情况下也能工作，只是不够健壮。

✅ "[不变量] b->valid == 1 意味着 b->data 对应当前的 b->dev 和 b->blkno。
   如果这条被破坏，bread() 会把块 A 的数据当成块 B 返回给调用者——不是可能出错，
   是一定会出错，只是取决于调用者什么时候用到那个错误数据。"
```

### 假模式 2：只找出不变量但不找破坏点

```
❌ "[不变量] 锁必须在使用共享数据前获取。这条不变量被破坏了。"
   → 说了等于没说。关键是：在什么环节？哪个具体代码行？

✅ 必须走到 Step3——列出所有潜在破坏点，然后通过验证逐一排除，最终锁定一个。
```

### 假模式 3：找到了不变量就停，不加断言

```
❌ "根因找到了，就是 bget 没清零 valid。"
   → 找到不变量被破坏的位置是一件事，防止它再次被破坏是另一件事。

✅ Step4 追问："如果当时代码里有一行 assert(b->valid == 0 || (b->valid == 1 && ...)),
   这个 bug 会在一开始就被发现吗？如果不会——为什么这个断言加不上？
   如果能加上——为什么不加？"
```

### 假模式 4：不变量太宽泛 —— "系统必须正确工作"

```
❌ "[不变量] 文件系统必须正确返回数据。"
   → 这个不变量没有提供任何 executable 的检查点。

✅ 不变量必须是可编码为 assert 的布尔表达式。
   "bread(dev, blkno) 返回的 buffer->data 必须等于磁盘上 (dev, blkno) 处的内容"
   可以被检查（比较 buffer 内容和重新从磁盘读的内容），
   "文件系统必须正确"不能被检查。
```

## 不同领域的示例

### 示例 1: OS 内核 — Buffer Cache（延续 EXT4 案例）

```
[不变量·Step1]
不变量：b->valid == 1 ⇒ b->data 对应当前的 (b->dev, b->blkno)
后果：bread() 返回错误块内容 → 用户拿到错误机器码 → page fault

[不变量·Step2]
维护者：bget() —— 分配/复用 buffer 时维护
bread() —— 读取磁盘后设 valid=1
brelse() —— 不应该改变 valid（除非是 invalidate 操作）

[不变量·Step3] 破坏点：
1. bget() 复用路径：更新了 dev/blkno 但没清零 valid ← 当前根因
2. bget() 分配新 buffer：如果 slab 分配器返回了未清零的内存，valid 可能是脏的
3. 外部 DMA：如果有人直接往 buffer->data 写数据而不经过 bread，valid 标志失去意义

[不变量·Step4] 可加检查：
在 bread() 开头：assert(b->valid == 0 || (b->dev == dev && b->blkno == blkno));
在 bget() 复用路径：如果 dev 或 blkno 变了 → b->valid = 0;
```

### 示例 2: 数据库 — 连接池

```
[不变量·Step1]
不变量：conn->in_use == true ⇒ conn 被恰好一个事务持有
后果：两个事务共享一个连接 → 查询结果混在一起 → 数据泄露或错误

[不变量·Step2]
维护者：pool.get() —— 从池中取连接时设置 in_use
pool.release() —— 归还连接时清 in_use

[不变量·Step3] 破坏点：
1. 归还连接时忘记清 in_use → 连接永远不可复用（泄漏）
2. 异常路径（transaction 抛异常）没有 finally 块归还连接
3. 异步代码中回调引用了已归还的连接

[不变量·Step4] 可加检查：
在 pool.get() 返回前：assert(conn->in_use == false);
在 pool.release() 中：assert(conn->in_use == true); conn->in_use = false;
```

### 示例 3: 前端 — 受控组件

```
[不变量·Step1]
不变量：受控 input 的 value prop 必须和 React state 中存储的值相同
后果：用户输入被覆盖（props 传到旧值）或有内容无法被编辑（state 没更新）

[不变量·Step2]
维护者：onChange handler —— 用户输入时更新 state
父组件 —— 传递 value prop

[不变量·Step3] 破坏点：
1. onChange 处理函数中用了过期的闭包变量 ← 常见 bug
2. 父组件在 shouldComponentUpdate 中错误地返回了 false
3. key prop 缺失，React 复用了错误的组件实例

[不变量·Step4] 可加检查：
在 render 中加 console.assert(props.value === state.value,
  "Controlled input invariant violated");
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 这条不变量可以被编码为一个 assert 吗？ | 可以写成布尔表达式 | 不变量含"应该""尽量""大多数情况下" |
| 如果这条不变量被破坏，我能说出具体后果吗？ | 能说出"哪个操作会得到什么错误结果" | 后果只有"系统会出错"或"可能有 bug" |
| 我是不是只找到了一个破坏点就停了？ | 列了所有可能破坏点 | 找到一个就宣布"就是它了" |

## 转换信号

### 何时进入
- [溯因推理] 锁定了根因方向 → 用不变量精确定位根因
- 审查代码时 → 先找不变量，再找破坏点
- 设计新模块时 → 先列不变量，再写代码
- 发现一个"不可能发生"的现象 → 必然有不变量被破坏了

### 何时退出
- Step3 列出了所有破坏点且已通过 [实践论] 定位到具体一处 → 转 [收束] 锁定
- 找不出完整的不变量（前提条件不清楚） → 回到 [现象学观察] 或 [格物致知] 读源码
- 发现不止一条不变量被破坏 → 转 [还原论] 分别分析各条不变量

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [溯因推理] 锁定了大致方向 | **不变量思维** | [收束] 单一锁定 + [实践论] 修复验证 |
| [整体观] 先知道哪个模块的不变量被破坏 | | [时间性推理] 为什么之前没有触发这个不变量被破坏 |
| [还原论] 每个模块有自己的不变量，分开分析 | | [辩证法] 回看：这条不变量为什么没被写下来 |

## 标注前自检清单

- [ ] 我列出的不变量能被写成 assert 吗？
- [ ] 我找到了所有潜在破坏点（不止一处）吗？
- [ ] 我能否精确描述"不变量被破坏后的具体后果"？
- [ ] 我是否建议了一个显式的断言/检查来防止它再次被破坏？

**任何一条未通过，不变量分析没有完成。**
