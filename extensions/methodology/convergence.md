# 收束思想

## 触发条件

- 面对一个开放性问题，可能性空间太大
- 排查了多个方向但不确定是否遗漏
- 讨论发散，话题漂移
- 任何问题解决过程——收束是 Debug/分析/设计的通用底层逻辑

## 核心思想

收束不是"缩小范围"这么简单。它是一个有结构的思维操作：**在每一步都明确知道可能性空间的边界在哪里，然后用证据去压缩它。**

不收束的代价：排查了 5 个方向，但不知道这 5 个方向是否覆盖了所有可能。排除 4 个，还剩几个？不知道——因为你没画过可能性空间的边界。

## 收束四步法

### 1. 划界：先画出可能性空间

在开始排查之前，先问：**所有可能的原因/方案/方向有哪些？**

```
列出全部候选方向。每个方向必须满足：
- 互斥：方向之间不能重叠
- 穷尽：所有可能的方向都列出来了
```

如果列不出穷尽的列表，至少列出你已知的，然后标注一个 `[未知]` 桶——"还有我们没想到的可能性"。

### 2. 标验证：每个方向对应一个验证方式

每一个候选方向必须有对应的验证方式。如果某个方向"不知道该怎么做"，那就是当前排查的瓶颈——优先设计验证实验。

```
候选 A → 验证方式 A' → 如果排除，可能性空间缩小到 B/C/D
候选 B → 验证方式 B' → 如果排除，可能性空间缩小到 A/C/D
```

验证方式必须是具体操作：加一行代码、跑一个命令、查一个寄存器——不能是"想想看"。

### 3. 逐向排除：每次排除，显式宣布收束

每排除一个方向，必须停下来确认：

> "这个方向排除了。新的可能性空间：B / C / D。还剩 3 个方向。接下来验证 B，因为 [最短验证路径 / 最高概率 / 最便宜]。"

不显式宣布收束，你就不知道排查进展到了哪一步。

### 4. 单一锁定：当只剩一个可能

当所有其他方向都被排除，剩下的那个就是根因。此时做最后一步确认：

> "所有线索指向 X。有没有任何现象不能被 X 解释？如果有——回到划界，你漏了某个方向。"

## 收束的五个常见错误

| 错误 | 表现 | 纠正 |
|------|------|------|
| **未划界就开始排查** | 查了三个方向，不知道还剩几个 | 先画可能性空间，再排查 |
| **验证不彻底就宣布排除** | "大
概不是这个原因" | 每个方向的排除必须有明确证据 |
| **排除后不宣布收束** | 排查推进了但意识还留在上一步 | 每排除一个方向，显式说"可能性空间缩小到 X" |
| **忘了整体观** | 只在一个层排查，忽略了上层或下层 | 可能性空间必须覆盖所有相关层 |
| **锁定后不做最终检验** | 找到一个解释就宣布胜利 | 用锁定的原因反查所有现象——是否存在被漏掉的反例？ |

## 和七步法的关系

收束思想贯穿整个七步法，但在以下几步特别关键：

- **摊开 → 划界**：摊开不是为了铺得越大越好，是为了画出可能性空间的边界
- **假设 → 标验证**：每个假设必须自带验证方式，否则只是猜测
- **排除 → 逐向排除**：排除一个方向后显式宣布剩余空间缩小
- **锁定 → 单一锁定**：所有线索指向同一处，最终反查

## 对话中的使用方式

以 `[收束]` 标注。每次收束动作都要让对话双方明确意识到可能性空间的变化：

> `[收束]` 现在排查了三个方向：A（排除了——验证实验证明不是）、B（排除了——extent 只有一条）、C（待验证——疑似 cache 不变量问题）。可能性空间从 4 个缩小到 2 个。接下来验证 C。

> ⚠️ 收束不是"缩小思考范围"——是"在明确标注的边界内，用证据逐个排除"。没画过边界，就不存在收束。
