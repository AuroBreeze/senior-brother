# 收束思想

## 这是什么

收束不是一个动作，是一组有结构的操作：**在每一步都知道可能性空间的边界在哪里，然后用证据去压缩它。** 排查了 5 个方向但不知道是不是覆盖了所有可能——这不是排查，这是随机游走。真正的收束要求你每一次排除方向后显式宣布"可能性空间从 X 个缩小到 Y 个，接下来验证 Z"。

> 收束的反面不是发散，是不知边界。发散本身没问题——但每个方向的进出必须有明确标记，否则你只是在可能性的海里乱游。

## 四步操作流程

### Step 1: 划界 —— 画出可能性空间

在开始排查之前，枚举所有可能的候选方向。方向之间必须互斥（不能重叠），尽量穷尽（不能有"其他"吞掉一半可能性）。如果穷尽不了，必须标注 [未知] 桶。

```
[收束·Step1] 可能性空间：
A: ______（验证方式：______）
B: ______（验证方式：______）
C: ______（验证方式：______）
D: ______（验证方式：______）
[未知]：______（还有什么我们可能没想到的？）
```

> 如果某个方向"不知道该怎么做验证"——这就是当前排查的瓶颈。优先设计验证实验，而不是跳过这个方向。

### Step 2: 排验证优先级

不是先验证最有可能的——是先验证最快能排除的。一个方向越快能排除，越应该先验证。

```
[收束·Step2] 验证顺序：
1. 先验证 ______：验证成本最低（______ 分钟 / ______ 行代码），排除后空间缩小到 {____}
2. 再验证 ______：成本中等，排除后只剩 {____}
3. 最后验证 ______：成本最高，但之前的方向已经排除了大部分可能性
```

### Step 3: 逐向排除 —— 每次必须显式宣布

每排除一个方向，必须说三件事：排除了什么、为什么（证据）、还剩什么。

```
[收束·Step3]
❌ 排除了 ____。证据：______（具体实验结果/观测）。
   可能性空间从 {____} 缩小到 {____}。还剩 ____ 个方向。
```

> 不宣布收束的排查 = 你排查了 3 个方向，但脑子还停留在第一个方向。显式宣布是为了让你自己知道进展到了哪一步。

### Step 4: 单一锁定 + 最终反查

当只剩一个方向时，不是宣布胜利——是反查：用锁定的原因去解释所有现象，有没有解释不了的？

```
[收束·Step4]
🔒 锁定：______（根因）。
   反查：这个原因能解释最初的所有现象吗？
   - 现象 1：____ → 能解释，因为 ____
   - 现象 2：____ → 能解释，因为 ____
   - 现象 3：____ → 能解释，因为 ____
   是否有任何现象不能被此原因解释？____（有 → 回到 Step1，可能漏了某个方向）
```

## 五种假执行模式

### 假模式 1：没划界就开始排查

```
❌ 查了三个方向（exec 参数、用户栈、extent 映射），觉得差不多了。
   但没问过：还有没有别的可能？block cache？磁盘驱动？编译器？

✅ 先画可能性空间。"调用链上 5 个可能的出错层：exec → VFS → EXT4 → cache → 磁盘。"
   然后逐个排除。每排除一个，显式宣布空间缩小。
```

### 假模式 2：验证不彻底就宣布排除

```
❌ "extent 映射应该没问题，因为它只有一个 extent。"
   → "应该"不是证据。

✅ "验证：debugfs stat 显示 busybox 只有一条连续 extent
   (逻辑块 0..338 → 物理块 33415..33753)。
   结论：ext4_read_file 不会做块拼接，exclude EXT4 extent 层。"
   有命令、有输出、有推理的排除才是真的排除。
```

### 假模式 3：剩多个方向时没有优先级

```
❌ "还剩 B 和 C，再看看。"
   → "再看看"不是计划。

✅ "还剩 B（缓存不变量）和 C（磁盘驱动）。先验证 B 因为：
   增加一行 assert 即可（5 秒改代码），而验证 C 需要加 tracepoint 到驱动层。
   如果 B 排除了再验证 C。"
```

### 假模式 4：锁定后不做反查

```
❌ "找到了！bget 没清零 valid。问题解决了。"
   → 没有反查：这个原因能解释"错误数据和正确数据刚好差 224 个页"吗？

✅ "反查：bget 复用了之前读过的 buffer，所以错误数据是之前某个块的内容。
   偏移差 224 页 = 这个 buffer 之前被用来读了块 200，现在被复用来读块 424。
   差值等于之前读过的块和当前需要的块之间的距离——完全吻合。"
```

### 假模式 5：把 [未知] 桶当垃圾桶

```
❌ "[未知]：可能是其他原因。"
   → "其他"没有信息量。

✅ "[未知]：我们还没考虑过的可能性——
   ① 中断处理程序在 I/O 过程中修改了页表？
   ② QEMU 的 virtio 模拟有 bug？（概率低但不能排除）
   ③ 某个我们没有读过的内核路径也在操作这块内存？"
```

## 不同领域的示例

### 示例 1: 后端 — 订单状态不一致

```
[收束·Step1] 可能性空间：
A: 数据库事务隔离级别不够 → 验证：检查 isolation level 配置
B: 应用层两次更新之间未加锁 → 验证：检查代码中的 @Transactional 注解
C: 消息队列重复消费 → 验证：检查 consumer 的 offset commit 日志
D: 缓存和数据库不同步 → 验证：检查 Redis 中订单状态和 DB 对比
[未知]：外部回调（支付网关）修改了订单状态？

[收束·Step2] 验证顺序：
先验证 A（查配置，10 秒）→ 再验证 B（读代码，2 分钟）→ 再验证 D（查日志，5 分钟）→ 最后 C（需要重现）

[收束·Step3]
❌ 排除了 A：isolation=READ_COMMITTED，对 UPDATE 有行锁。
   空间从 {A,B,C,D} 缩小到 {B,C,D}。

❌ 排除了 B：所有订单更新都在同一个 @Transactional 内。
   空间缩小到 {C,D}。还剩 2 个方向。

[收束·Step4]
🔒 锁定 C（消息重复消费）：consumer 在处理完后 offset commit 超时，
   消息被 rebalance 到另一个 consumer 重新处理。
   反查：解释了为什么只有高负载时才出现（超时），以及为什么总是"多扣一次库存"。
```

### 示例 2: 前端 — 页面偶发白屏

```
[收束·Step1]
A: JS 报错导致 React 树卸载 → 验证：console 错误日志
B: 某个 chunk 加载失败 → 验证：Network 面板 + chunk load error
C: CSS 被错误注入 → 验证：检查 body 是否有 display:none
D: 第三方脚本阻塞渲染 → 验证：Performance 面板
[未知]：浏览器扩展干扰？

[收束·Step2]
先验证 A（打开 console，即时）→ 再验证 B（Network 面板，1 分钟）→ 再验证 C（DOM 检查，1 分钟）

[收束·Step4]
🔒 锁定 B：CDN 偶尔返回 404 导致 JS chunk 加载失败，React lazy 边界没处理好 fallback。
   反查：解释了为什么只白屏不报错（lazy 边界吞了错误），以及为什么刷新就好了（CDN 下次返回 200）。
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 我画出可能性空间了吗？ | 有互斥+接近穷尽的列表 | 方向之间有重叠，或有巨大的 [未知] 桶 |
| 我每排除一个方向都显式宣布了吗？ | 每次排除有三句话（排除+证据+剩余） | 默默排查，没有收束宣言 |
| 锁定后我反查了所有原始现象吗？ | 每个现象都被解释了 | 有一些现象"没关系"或"先不管" |

## 转换信号

### 何时进入
- 任何排查/分析开始 → Step1 划界
- 线索太多，不知道从哪下手 → 收束给方向排序
- 排查了一段时间但没有进展感 → 你可能没有在收束（没有宣布过排除）

### 何时退出
- Step4 锁定 + 反查通过 → 转 [辩证法] 回看和 [实践论] 修复验证
- 反查失败（有一个现象解释不了）→ 回到 Step1，可能性空间漏了一个方向
- 所有方向都排除了但没有锁定 → 可能性空间划错了，回到 [整体观] 重新看全局

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [现象学] 事实清单 → 划界 | **收束思想** | [辩证法] 回看 |
| [溯因] 形成假设 → 作为候选方向 | | [实践论] 执行排除验证 |
| [整体观] 确保可能性空间覆盖所有层 | | [不变量] 锁定后精确定位 |

## 标注前自检清单

- [ ] 可能性空间是否画出来了（互斥 + 接近穷尽）？
- [ ] 每个方向是否有对应的验证方式？
- [ ] 是否有 [未知] 桶且至少写了 2 个可能的未知方向？
- [ ] 最近一次排除是否显式宣布了"空间从 X 缩小到 Y"？
- [ ] 如果已锁定——是否做了反查？

**收束的核心纪律：不画边界不排查，不宣布不排除。**
