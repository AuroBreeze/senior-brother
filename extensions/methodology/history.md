# 历史方法

## 这是什么

不是"了解一下历史背景"。历史方法是一个具体的理解策略：**不从当前形态出发，从最初版本出发——展示一个简单的东西如何在每个阶段被新需求推着演化，变得复杂。** 看懂了演化过程，自然能区分"哪些是必要复杂性、哪些是历史遗留"。

> 核心洞察：很多复杂系统不是"设计出来的"，是"演化出来的"。如果你只看最终形态，你会觉得它设计得很怪——但如果你追溯它的演化路径，你会发现它一开始也很朴素，每个阶段的添加在当时都是合理的。当前形态里的混乱不是某个人设计的，是演化路径留下的。

## 三步操作流程

### Step 1: 追溯起点——它最初解决什么问题？

退到系统最早、最简单的版本。它最初是为了解决一个什么具体问题？

```
[历史方法·Step1]
最初问题：______（具体场景/痛点）
最初方案：______（一句话——最初的版本有多简单？）
最初不解决的问题：______（哪些现在看起来理所当然的功能最初没有？）
```

### Step 2: 追溯演化路径——每个阶段加了什么？为什么？

找出关键的转折点。每个阶段的添加回答：什么新需求/新约束推动了这次变化？

```
[历史方法·Step2]
阶段 1（____年）：______（最初的简单形态）
阶段 2（____年）：因为 ______（新需求/新约束），加了 ______
阶段 3（____年）：因为 ______，加了 ______
...
当前形态：______
```

### Step 3: 区分——哪些是必要复杂性，哪些是历史遗留？

```
[历史方法·Step3]
必要复杂性（如果去掉，一个真实的需求会无法满足）：
  - ______：因为 ______ 需要它
  - ______：

历史遗留（当初合理现在尴尬，但去掉成本太高）：
  - ______：当初是为了 ______，但现在 ______。
    去不掉的因为 ______（向后兼容/生态依赖）
  - ______：
```

## 四种假执行模式

### 假模式 1：只查历史不分析结构

```
❌ "[历史方法] git checkout 既是切换分支又是恢复文件，
   因为在最初的设计中分支切换就是覆盖工作树。"
   → 说了历史，但没区分这是必要复杂性还是历史遗留。讲了一个故事——然后呢？

✅ 必须走到 Step3——"这个双重语义是历史遗留。如果重新设计 git，
   会用 switch（安全切换）和 restore（恢复文件）两个命令——事实也如此，
   git 2.23 确实加了这两个命令。旧的 checkout 保留了以避免破坏脚本。"
```

### 假模式 2：把演化路径当成辩护——"因为它一直是这样的"

```
❌ "[历史方法] 这个接口设计之所以奇怪，是因为它在 2005 年就是这样定义的。"
   → "一直是这样的"不是理解——是投降。

✅ 历史方法告诉你"为什么变成了这样"，不替你判断"这样好不好"。
   演化路径解释当下，不等于辩护当下。
```

### 假模式 3：只往回看，不往前看

```
❌ 分析了演化路径，觉得理解了。结束。
   → 看演化路径的价值之一是预测：如果当前趋势继续，下一步会往哪走？

✅ "如果演化趋势继续（每 2 年加一层抽象来应对新增需求），
   下一个版本可能需要 ______。或者——现在是时候重新设计而不是再加一层了。"
```

### 假模式 4：用自己的假设编历史

```
❌ "[历史方法] 这个设计应该是先有 A，后来觉得不够加了 B。"
   → "应该是"不是历史方法——你在用想象力填补历史。

✅ 历史方法需要证据：commit log、设计文档、mailing list 讨论、changelog。
   如果没有证据，标注 [猜测]，不说 [事实]。
```

## 不同领域的示例

### 示例 1: 编程语言 — C 的数组退化

```
[历史方法·Step1]
最初问题：在内存非常有限（几十 KB）的机器上实现 C 编译器。
最初方案：数组名就是指向第一个元素的指针——不需要额外的数组元数据（长度、类型信息）。
最初不解决：数组越界检查、数组长度获取。

[历史方法·Step2]
阶段 1：B 语言把数组实现为指针，C 继承了这个设计。
阶段 2：引入 struct——struct 包含数组时可以内联存储，不退化。
阶段 3：引入 sizeof 操作符——但 sizeof(array) 返回数组总大小，
        而 sizeof(pointer) 返回指针大小。不一致性的根源埋下。
阶段 4（C99）：引入 VLA（变长数组），sizeof 动态计算——但这个特性后来被各编译器视为可选。
阶段 5（C11/C17）：VLA 降级为 optional。数组退化的根本问题从未解决。

[历史方法·Step3]
必要复杂性：数组退化为指针使得函数传参不需要复制整个数组（性能关键）。
历史遗留：sizeof 对数组和指针的不一致行为——当初合理（编译器需要知道数组大小来分配），
但现在导致大量"array vs pointer"的困惑。
去不掉的原因：几乎所有现存的 C 代码都依赖数组退化为指针的行为。
```

### 示例 2: Web — CSS 的垂直居中

```
[历史方法·Step1]
最初问题：HTML 是为文档（论文、报告）设计的，不是为应用 UI 设计的。
最初方案：CSS 处理"流式布局"——元素从左到右、从上到下排列。
最初不解决：垂直居中（在文档布局中不需要——段落不居中对齐）。

[历史方法·Step2]
阶段 1（CSS1, 1996）：没有垂直居中。人们用 table 布局（HTML 的 table 天然支持垂直居中）。
阶段 2（CSS2, 1998）：加了 vertical-align: middle——但只在 table-cell 和 inline 上下文中有效。
阶段 3（2000s）：人们发现一堆 hack——display:table, position:absolute+负margin, line-height hack。
阶段 4（CSS3, 2010s）：Flexbox（align-items: center + justify-content: center = 真正的居中）。
阶段 5：Grid（place-items: center = 一行代码居中）。

[历史方法·Step3]
必要复杂性：Flexbox 和 Grid 的轴概念（main axis vs cross axis）是真正的必要复杂性——
        一维和二维布局的本质差异。
历史遗留：vertical-align 在非 table 上下文中的行为——当初不是为垂直居中设计的，
        名字误导了一代开发者。去不掉因为向后兼容。
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 演化路径中的每个阶段是否因果可追溯（新需求→新特性）？ | 每个阶段有具体触发原因 | 只是时间顺序罗列，没有因果链 |
| 我是否区分了必要复杂性和历史遗留？ | 至少各列出一条 | 只说历史没有区分 |
| 我的历史追溯是否有证据来源？ | 有来源；无来源时标 [推测] | 用"应该是""可能是"编历史 |

## 转换信号

### 何时进入
- 理解一个看起来"过度设计"的复杂系统
- 用户说"这个设计好奇怪，为什么要这样？"
- [第一性原理] 的 Step5 发现实际实现和最小方案差距很大 → 历史方法解释差距
- 学习新技术时感到不知所措（跟踪演化路径降低认知负担）

### 何时退出
- Step3 必要复杂性 vs 历史遗留区分完成 → 可以进入局部细节分析
- 发现演化路径中有理解缺口 → 转 [格物致知] 查设计文档/commit log
- 历史遗留部分如果想清理 → 转 [收束] 划可行性边界

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [第一性原理] Step5 发现设计比最小方案复杂很多 | **历史方法** 解释复杂性是怎么演化出来的 | [结构主义] 分析当前的关系网络 |
| [整体观] 先知道当前结构 | | [辩证法] 演化中的矛盾推动力 |

## 标注前自检清单

- [ ] 是否从最初版本出发，展示了完整演化路径？
- [ ] 每个演化阶段是否有具体的新需求/新约束作为推动力？
- [ ] 是否区分了必要复杂性和历史遗留（至少各一条）？
- [ ] 如果没有确切历史证据，是否标了 [推测]？

**复杂系统不是设计出来的，是演化出来的。看懂演化路径，复杂就变得可理解。**
