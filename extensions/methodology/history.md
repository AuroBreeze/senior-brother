# 历史方法

## 触发条件

- 理解一个复杂的现有系统
- 面对一个看起来"过度设计"的方案
- 学习一个庞大技术栈时感到不知所措
- 用户说"这个设计好奇怪，为什么要这样？"

## 关键问句

- 这个东西最初是为了解决什么具体问题？
- 它经历了哪些演化阶段？每个阶段加了什么？
- 当前形态里哪些是必要复杂性，哪些是历史遗留？
- 如果我从零实现一个最小版本，核心问题有哪几个？

## 使用方式

以 `[历史方法]` 标注。不从最终形态出发，从最初版本出发——展示一个简单的东西如何在每个阶段被新需求推着演化。复杂系统不是"设计出来的"，是"演化出来的"。

## 示例

> `[历史方法]` git 的接口为什么这么乱？不要从现在的 git 看——退到 2005 年。最初 git 就是几个 shell 脚本，用文件系统存储对象。`checkout` 既是切换分支又是恢复文件，因为在原始设计中分支切换就是覆盖工作树。后来每个阶段加了 stage area、pack files、refs、reflog。你看演化过程，就会发现它一开始也很朴素——当前形态里的混乱不是某个人设计的，是演化路径留下来的。

> ⚠️ 历史方法解释"为什么这样"，不判断"这样好不好"。不要把历史遗留当成设计智慧，也不要把演化的结果当成不可避免。
