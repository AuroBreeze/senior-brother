# 还原论拆解

## 这是什么

不是"把大问题拆成小问题"。还原论是一个具体的操作：**把系统拆成多个可以独立验证的部件，每个部件有明确的输入/输出，然后逐个验证——验证完后再确认拼回去行为是否一致。** 关键是"可独立验证"——拆出来的每个部分必须能在不依赖其他部分结果的情况下被证实或证伪。

> 区分："先查 A，再查 B"不一定是还原论。还原论要求你拆解后每个部件的正确性可以单独判断，部件之间通过明确的接口连接。如果 A 的结果完全取决于 B 的结果，你没有拆开——你只是在同一个问题上走了两遍。

## 四步操作流程

### Step 1: 确定拆解粒度

沿着数据流/调用链，找出每个有明确输入和输出的环节。拆到每一层有明确的"进去什么、出来应该是什么"。

```
[还原论·Step1] 拆解（沿数据流）：
输入：______
  → 环节 1：______（输入：____, 输出：____, 正确性判据：____）
  → 环节 2：______（输入：____, 输出：____, 正确性判据：____）
  → 环节 3：______（输入：____, 输出：____, 正确性判据：____）
输出：______（最终结果）
```

> 拆到每个环节的正确性判据可以用一句话说清。"应该正确"不是判据。

### Step 2: 标依赖

标注环节之间的依赖关系。哪些可以并行验证？哪些必须串行（后一个依赖前一个的正确输出）？

```
[还原论·Step2] 依赖关系：
环节 1 ← 无依赖，可独立验证
环节 2 ← 依赖环节 1 的输出 ____（如果环节 1 错了，环节 2 的验证无意义）
环节 3 ← 依赖环节 2 的输出 ____
```

### Step 3: 确定验证顺序

从最底层（最靠近输入）开始验证。底层如果错了，上层必然错——先验证底层可以避免在上层浪费时间。

```
[还原论·Step3] 验证顺序：
先验证环节 1（最底层，无依赖）→ 再验证环节 2 → 最后验证环节 3
或者：环节 1 和环节 3 无依赖关系 → 可并行验证
```

### Step 4: 逐个验证 + 整体复原

每个环节验证完后，在它被确认正确的条件下验证下一环。全部验证完后，确认整体行为在全局中解释得通。

```
[还原论·Step4]
环节 1：验证结果 ____ → √/✗
环节 2：验证结果 ____（基于环节 1 正确的输入假设）→ √/✗
环节 3：验证结果 ____ → √/✗
整体复原：所有环节的结论在全局中是否自洽？____
```

## 四种假执行模式

### 假模式 1：拆了但没有独立验证

```
❌ "[还原论] 这个问题可以拆成 exec、VFS、EXT4、block cache 四层。"
   → 拆了，但每层没有独立验证方式。"先看 exec 对不对"——怎么看？

✅ 每层必须带判据。"验证 exec：检查 ELF LOAD 段的 p_offset 和 p_filesz
   是否被正确传递给 readi。在 exec 中加 printk 打印这两个值，
   和 readelf 的输出对比。如果一致→exec 层正确，问题在下游。"
```

### 假模式 2：拆得太粗——一个环节等于整个问题

```
❌ "问题可以拆成：代码问题 vs 配置问题 vs 环境问题。"
   → 这不是拆解，这是分类。分类不提供验证路径。

✅ 必须沿数据流/调用链拆。每一层有具体的输入输出。
   "exec(path) → loadseg(ELF) → readi(inode, offset) → ext4_read_file() → bread(dev, blkno)"
```

### 假模式 3：拆完忘了装回去

```
❌ "经过验证，bread 层有问题。问题就在缓存。"
   → 确认了局部问题，但没确认这个局部问题是否能完全解释全局现象。

✅ 装回去："bread 返回了错误数据→ext4_read_file 拿到错误数据→readi 拿到错误数据
   →loadseg 把错误数据写入用户页表→用户执行错误机器码→page fault。每一环都吻合。"
```

### 假模式 4：跳过依赖分析，从中间开始验证

```
❌ "先看看 ext4_read_file 对不对。"
   → 如果上游 readi 传了错误的 offset，ext4_read_file 正确也没用。

✅ 从依赖链的最底层开始验证。没有依赖的环节优先验证。
```

## 不同领域的示例

### 示例 1: 后端 — 用户注册慢

```
[还原论·Step1] 拆解（沿数据流）：
HTTP Request →
  环节 1: 参数校验（输入：JSON body, 输出：validated fields / 400, 判据：<5ms）
  环节 2: 密码哈希（输入：password, 输出：bcrypt hash, 判据：<200ms 单个）
  环节 3: DB INSERT（输入：user record, 输出：auto_increment id, 判据：<50ms）
  环节 4: 发送验证邮件（输入：email, 输出：SMTP accepted, 判据：<500ms）
  环节 5: 返回响应

[还原论·Step2] 依赖关系：
环节 1,2 无依赖 → 可并行
环节 3 依赖 2（需要 hash 后的密码）
环节 4 依赖 3（需要 user.id 放入邮件链接）
环节 4 可以和 5 并行（异步发邮件）

[还原论·Step4]
环节 1: 1ms ✓
环节 2: 450ms ✗ ← bcrypt cost factor 设成了 14，降到 10 应该是 ~80ms
环节 3: 30ms ✓
环节 4: 发现 4 是同步等待 SMTP 的 → 改为异步，不阻塞注册响应
```

### 示例 2: 前端 — 页面首屏加载慢（LCP 3.5s）

```
[还原论·Step1] 拆解（沿加载时间线）：
1. DNS + TCP + TLS（输入：URL, 输出：connection established, 判据：<300ms）
2. HTML 下载 + 解析（输入：HTTP response, 输出：DOM, 判据：<200ms + 解析）
3. 阻塞资源加载（CSS + 同步 JS）（输入：HTML 中的 <link>/<script>, 输出：loaded resources, 判据：<500ms 总计）
4. React 水合（输入：SSR HTML + JS bundle, 输出：interactive page, 判据：<500ms）
5. LCP 图片加载（输入：hero image URL, 输出：rendered image, 判据：剩余时间）

[还原论·Step2]
环节 1→2→3→4 串行依赖。环节 3 内部：CSS 和 JS 可以并行下载但 JS 执行阻塞 CSS 解析。
环节 5 可以和 4 并行（图片不阻塞水合）。

[还原论·Step4]
环节 3: 发现有一个 500KB 的第三方 SDK 同步加载 → 改为 async/defer → 节省 800ms
环节 5: LCP 图片用 <img loading="lazy"> → 改为 fetchpriority="high" + preload → 节省 400ms
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 每个环节有独立的正确性判据吗？ | 每个环节都有具体的"进去什么、出来应该是什么" | 判据是"环节 X 是否正确"——这等于没给判据 |
| 我是否从最底层（无依赖的）开始验证？ | 验证顺序遵循依赖拓扑序 | 跳过了底层直接查中间层 |
| 验证完后是否确认了全局自洽？ | 所有环节的结论能串联成完整因果链 | 局部有问题但没确认这个局部问题是全局唯一原因 |

## 转换信号

### 何时进入
- 问题涉及多个组件/模块，不能一眼看到根因 → 还原论拆解
- 排查时感到"问题太大了不知道怎么下手" → 拆到可以下手的大小
- 一个复杂系统需要理解 → 从输入到输出逐层追踪

### 何时退出
- 每个环节已验证完毕且全局自洽 → 转 [收束] 锁定或 [实践论] 做端到端验证
- 发现某个环节无法独立验证（输入输出不明确）→ 这个环节本身需要进一步拆解
- 拆解中发现了新的未知 → 转 [格物致知] 读源码/[现象学] 收集事实

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [整体观] 先知道全貌（调用链/数据流），还原论在这个基础上拆 | **还原论拆解** | [整体观] 拆完验证完，装回去确认全局自洽 |
| [现象学] 事实不够时先收集，不要盲拆 | | [溯因] 某个环节有问题，溯因定位该环节内的根因 |
| | | [实践论] 对每个环节执行验证实验 |

## 标注前自检清单

- [ ] 是否沿着数据流/调用链拆解（不是按概念分类）？
- [ ] 每个环节是否有独立的正确性判据（可以用一句话说清）？
- [ ] 是否标注了环节间的依赖关系？验证顺序是否从无依赖的底层开始？
- [ ] 全部验证完后是否做了整体复原（确认局部结论能拼解释全局现象）？

**未拆到每个环节有独立判据的程度 → 继续拆。**
