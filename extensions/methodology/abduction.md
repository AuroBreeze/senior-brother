# 溯因推理

## 这是什么

溯因不是演绎（从规则+前提推出必然结论），不是归纳（从多样本总结规律）。溯因是：**面对一组令人困惑的现象，找出能最简洁、最自然地解释所有现象的那个假设。** 结论不是"必然为真"，而是"当前最佳解释"——必须用实验验证后才能升级。

> 区分：演绎="所有人都会死，苏格拉底是人，所以苏格拉底会死"（前提为真则结论必为真）。归纳="我见过的 1000 只乌鸦都是黑的，所以乌鸦大概是黑的"（样本→概率）。溯因="草地上有湿的脚印，窗户开着，桌上的钱包不见了 → 有人从窗户进来了"（现象→最合理假设，需要验证）。

## 五步操作流程

### Step 1: 列出全部的、不可争议的现象

只放 [事实]，不放 [推断]。每个现象必须是通过观察/日志/寄存器 dump/实验结果可以直接确认的。

```
[溯因·Step1] 现象清单（仅 [事实]）：
1. ______（观察来源：______）
2. ______（观察来源：______）
3. ______（观察来源：______）
```

> 如果某个"现象"其实是你推断出来的——把它移到 Step2 的备注里，不要放在现象清单中。

### Step 2: 对每个现象问——"这个现象奇怪在哪？"

一个现象之所以需要解释，是因为它违反了某个预期。把这个预期显式写出来。

```
[溯因·Step2]
现象 1：______。我原本的预期是 ______。差距：______。
现象 2：______。我原本的预期是 ______。差距：______。
```

### Step 3: 生成最简解释

找一个能同时解释所有现象的单一原因。不是因为你能想到它，而是因为它需要的额外假设最少。

```
[溯因·Step3] 最能解释所有这些现象的最简解释是：______。
这个解释需要的额外假设：
  - ______（可独立验证？是/否）
  - ______（可独立验证？是/否）
额外假设总数：______
```

### Step 4: 推导可验证推论

如果这个假设成立，除了我们已经看到的现象，还应该能观察到什么？找一个当前的观察中不包含的新推论。

```
[溯因·Step4] 如果 Step3 的假设成立，那么还应该观察到：______。
验证方式：______（具体操作，不是"想想看"）。
验证结果如果是 X → 假设成立。如果结果是 Y → 假设被推翻。
```

> 这一步是溯因和瞎猜的分界线。没有可验证推论的假设不是溯因，是编故事。

### Step 5: 排出验证优先级

如果有多个假设，按验证成本从低到高排序。先验证最便宜、最容易推翻的。

```
[溯因·Step5]
假设 A（____个额外假设）：验证方式 ____，预计耗时/成本 ____
假设 B（____个额外假设）：验证方式 ____，预计耗时/成本 ____
先验证 ____，因为 ____。
```

## 四种假执行模式

### 假模式 1：现象没列全就开始解释

```
❌ "[溯因] 这个 bug 应该是缓存问题。"
   → 只看到了一个现象（数据不对），跳过了"为什么缓存的其他数据是对的？"
      等问题相关的现象没被纳入解释范围。

✅ 先把所有和问题相关的 [事实] 列完。如果一个假设解释不了某个事实，
   要么假设被推翻，要么那个"事实"其实是推断。两种结果都是进展。
```

### 假模式 2：只提一个假设，没有竞争假设

```
❌ "[溯因] 观察到的现象是 X，最合理的解释是 Y。"
   → 只有 Y，没考虑过 Z。这可能是你想到的第一个解释，不一定是合理的解释。

✅ 至少列出 2 个互斥假设（即使其中一个很弱）。"另一个可能是 ____，
   但它需要假设 ____（多了一个不必要的假设），所以优先级更低。"
   显式排除比默认忽略强。
```

### 假模式 3：解释能说通 = 解释被验证了

```
❌ "[溯因] 假设是缓存返回了错误数据。这解释了所有现象。所以就是它了。"
   → "能解释"不等于"被验证"。能解释所有现象的假设可能不止一个。

✅ 必须走 Step4——推导一个排他性的可验证推论。"如果缓存返回了错误数据，
   那么磁盘上 0x249fc 处的代码应该和用户内存中的错误代码完全一致。"
   如果一致→假设成立。如果不一致→即使"能解释"，也不是它。
```

### 假模式 4：把多个独立原因拼在一起

```
❌ "[溯因] 可能是缓存有个 bug，同时编译器也生成了奇怪的代码，
   而且磁盘可能有坏道。"
   → 三个独立原因同时发生的概率 = 三个概率的乘积。极低。

✅ 一个好的溯因解释是单一的、统一的。"原因 A 同时导致了现象 1 和现象 2"
   优于"原因 A 导致现象 1，原因 B 导致现象 2，它们恰好同时发生了"。
```

## 不同领域的示例

### 示例 1: 后端 — API 偶发超时

```
[溯因·Step1] 现象清单：
1. /api/search 端点 p99 延迟从 50ms 跳到 2000ms，每 15 分钟一次
2. 超时期间 CPU 使用率正常，内存正常，网络 IO 正常
3. 超时总是整点过 3 分钟开始，持续约 90 秒
4. 只有 search 端点受影响，其他端点正常

[溯因·Step2] 差距分析：
- 现象 1：预期是延迟稳定，实际有周期性尖峰
- 现象 3（时间规律）+ 现象 4（只影响 search）→ 不是全局资源问题

[溯因·Step3] 最简解释：某个定时任务（cron/后台作业）恰好访问了
与 search 共享的某个资源（连接池、索引锁、缓存过期），
每 15 分钟触发一次，耗时 90 秒。额外假设：1 个（存在这样的定时任务
且时间对齐）——极小。

[溯因·Step4] 可验证推论：如果假设成立，在超时期间应该看到：
- search 的数据库连接池全部被占用
- 或 search 依赖的某个缓存 key 批量过期
验证方式：在超时窗口抓数据库连接池状态和缓存命中率。

[溯因·Step5] 先查连接池（一行日志的事），再查缓存（需要加埋点）。
```

### 示例 2: 前端 — 组件渲染了两次

```
[溯因·Step1]
1. useEffect 的日志打印了两次，但 UI 看起来只渲染了一次
2. 只在开发环境出现，生产环境没有
3. 只在 StrictMode 包裹的组件中出现

[溯因·Step2]
- 生产环境正常但开发环境异常 → 不是逻辑 bug，是开发环境特有机制

[溯因·Step3] 最简解释：React 18 StrictMode 在开发环境故意
double-invoke effects 来检测副作用是否正确清理。
额外假设：0 个（这是文档里写明的行为）。

[溯因·Step4] 验证：关掉 StrictMode 看问题是否消失。
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 我的解释能同时覆盖所有现象吗？ | 每个现象都被解释了 | 有一个现象"先不管"或"可能是观测误差" |
| 我列出竞争假设了吗？ | 至少 2 个互斥假设 | 只有一个假设，没想过替代方案 |
| 每个假设带了可验证推论吗？ | 每个都有具体验证操作 | "验证方式：分析一下"——没有具体操作 |
| 我选择的假设额外假设是最少的吗？ | 是的，而且我知道排第二的是哪个 | 没数过额外假设数量 |

## 转换信号

### 何时进入
- 有多个现象需要统一解释 → 进入 Step1
- 你发现自己在说"这很奇怪"或"不应该这样" → 溯因开始
- [现象学观察] 收集了足够事实 → 自然转入溯因

### 何时退出
- Step5 排出优先级后 → 转 [实践论] 执行验证
- 验证后假设成立 → 如果解释的是 bug 根因，转 [不变量思维] 锁定
- 验证后假设被推翻 → 回到 Step1，新现象（实验失败）加入现象清单
- 发现多个假设都合理且无法区分 → 转 [奥卡姆剃刀] 排优先级

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| [现象学观察] 必须先收集现象——没事实就溯因 = 瞎猜 | **溯因推理** | [实践论] 验证推论 |
| [时间性推理] 时间规律本身可以是一个关键现象 | | [不变量思维] 假设验证通过后锁定 |
| [还原论] 如果系统太复杂，先拆再溯因 | | [奥卡姆剃刀] 多假设竞争时排优先级 |

## 标注前自检清单

- [ ] 现象清单全是 [事实]，没有混入推断或猜测？
- [ ] 我的解释是单一原因还是多原因的拼凑？
- [ ] 我是否数了额外假设的数量？
- [ ] 是否推导了至少一个可验证推论——如果我错了，实验会告诉我？
- [ ] 我是否考虑了至少一个竞争假设（哪怕只是显式排除）？

**任何一条未通过，不要宣称"溯因完成"。**
