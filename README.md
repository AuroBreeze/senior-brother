# 师兄 (Senior Brother)

> **认知示范 > 答案输出。** 答案只能解决一个问题，认知过程能解决一类问题。

**师兄**不是角色扮演，是一套**可执行的认知方法论**。它让 Claude 展示自己怎么从"不懂"走到"懂"——包括走错的路、卡住的点、排除过的假设、选择 A 放弃 B 的考量。你获得的不是正确答案，而是**下次自己也能找到答案的能力**。

## 为什么是现在

知识的获取成本正在趋近于零。**"知道什么"不再是竞争力。**

但大多数人用 AI 的方式是"提问 → 拿答案 → 复制粘贴"。这产生了一个悖论：解决的问题越多，**自己解决问题的能力越退化**。

更深一层：当 AI 能回答所有已知问题，真正的价值转移到了**"提出正确问题"和"从零到一探索未知"**——而这两种能力，恰恰在"直接拿答案"的使用模式中被闲置了。

LLM 默认是一个"答案输出者"——给你看似合理的回答，但你不清楚哪些是事实、哪些是猜测、推理在哪一步可以质疑。答案是对的，但**你不知道它是怎么来的**。

每次看到 AI 的回答，你自然会想：它为什么会这么做？它当时是怎么想的？再往下追问一层——**它的思维链是基于什么产生的？** 哪些是观察到的事实，哪些是推断，哪些是隐含假设？这一层决定了上面两层。

师兄做的就是把这层摊开。`[事实]` `[推断]` `[假设待验证]` `[猜测]` 四个标记让你看到每一步推理的认知根基，`>` 警告块标注隐含前提。你不用猜"它为什么会这么说"——草稿纸就在桌上。

师兄反过来：

- 不是给你答案，是让你看答案是怎么被找到的
- 不是替你思考，是让你看到思考可以是什么样子
- 不是节省你的认知，是**训练你的认知**

> 快速发展的时代，如果自身认知不能提速，就会被抛弃。知识可以被搜索，答案可以被生成，但**从混乱中找到结构、从现象中锁定根因、从零到一建立理解**——这些能力只能通过反复观看和实践认知过程来获得。

## 师兄怎么做

四个硬性动作，每条回复缺一不可。凭这四个特征你一眼能认出"这是师兄"：

| 动作 | 效果 |
|------|------|
| **`>` 警告块** | 标注隐含前提和易错点——"这个结论只在 X 条件下成立" |
| **认知状态标记** | `[事实]` `[推断]` `[假设待验证]` `[猜测]` ——确定和不确定，分开说 |
| **全局定位** | 每条回复第一句先告诉你"我们现在站在系统的哪个位置" |
| **回看沉淀** | 每次收束后追问"为什么这个坑会被踩到？下次怎么更快？" |

实际对话的感受是这样的：

> "不急，先把线索摊开。这个 bug 出在 bread/bget 这一层——它在整个 I/O 栈的这里："
>
> *[Mermaid 架构图展开调用链]*
>
> "[事实] readi 返回 0 时 block 未被读入。[推断] 根目录 inode 一定已在 itable 中。[假设待验证] 页面边界对齐导致分配失败。"
>
> "> ⚠️ 这里隐含了一个前提：M 态下 MMU 禁用了，SP 指向虚拟高地址访存会失败。"
>
> "这次我们学到了什么？这个经验可以提炼成一个检查项吗？"

## 什么时候用

**擅长：**
- Debug：从现象一步一步收束到根因，不是猜谜式修改
- 源码分析：从入口追到最底层，建立系统拓扑感
- 架构决策：追问"为什么选 A 不选 B？B 的代价你考虑过吗？"
- 学习新技术：展示"我是怎么学的"，而非知识点罗列

**不适用：** 非技术对话。

> **简单问题会被方法论压垮吗？** 不会。LLM 的默认倾向是偷懒——能短就不会长。面对"这个函数怎么用"之类的问题，模型仍然会简洁回答。方法论的四动作自带例外条款：纯确认、纯追问、非技术回复自动跳过。**方法论是地板，不是天花板——它只在模型最想偷懒的时候（复杂问题面前）拦住它走捷径。**

## 一个真实例子

以上是抽象描述。下面是一次完整 Debug 对话的压缩版，看这些动作在实战中怎么运转：

**用户：** "busybox 切换到 EXT4 后起不来，exec 进入用户态就 page fault。"

**师兄：** 不急着猜。先画调用链——`exec → loadseg → readi → ext4_read_file → bread → bget → virtio_disk_rw`，五个可能出错的层全部摊开。然后逐个假设，逐个验证。

验证一：反汇编 busybox 查 `_start` → busybox 自己从栈读 argc/argv，不依赖 kernel 传参 → **[排除] exec 参数**。

验证二：dump 用户栈 → argc、argv、auxv 全部正确 → **[排除] 栈布局**。

关键矛盾出现。epc 指向 `addi s0,s0,1`——纯寄存器运算，不访问内存。但 cause=13 是 load page fault。**addi 不可能触发访存。**

`[推断]` 用户内存中的机器码 ≠ 磁盘文件内容。probe 用户页表验证 → **确认**。objdump 搜索错误机器码 → 来自文件中另一个偏移，差值 = 224 个 4K 页 → `[推断]` 结构性错误，非随机位翻转。

验证三：debugfs 确认 extent 是连续单一块 → **[排除] EXT4 extent 映射**。

只剩 block cache。分析 `bread`/`bget` 逻辑：**bget 复用 LRU buffer 时更新了 dev/blkno，但没清零 valid**。bread 看到 valid=1，跳过磁盘读取，返回旧块内容。

`[锁定]` Buffer cache 不变量被破坏。修复：一行 `b->valid = 0;`。

`[回看]` EasyFS 阶段为什么不暴露？时间性推理：EasyFS 块少，buffer 复用几乎不发生；EXT4 339 个块，LRU 回收频繁触发。"切换文件系统 = 切换 cache 压力模式"。

七个步骤走完：摊开 → 分类 → 假设 → 验证 → 排除 → 锁定 → 回看。三次显式排除，一个矛盾发现，一条不变量锁定。

> 完整对话见 `skills/senior-brother/examples/ext4-debug.md`（579 行，含完整推理链 Mermaid 路线图）。

## 背后的认知框架

上面的七个步骤不是随意编排的。师兄背后是一套三层认知架构——"怎么看 → 怎么想 → 怎么推进"：

```
视域层（怎么看）              引擎层（怎么想）              收束层（怎么推进）
─────────────────          ─────────────────          ─────────────────
整体观      建立系统拓扑    溯因推理    反推最佳解释    现象学观察  先描述，不急解释
不变量思维  锁定被破坏的约束  第一性原理  退回不可拆的基本约束  还原论拆解  拆成最小可验证单元
时间性推理  为什么现在才出问题 格物致知    直接面对一手材料    实践论验证  最小实验说话
                          类比锚定    用日常直觉解释抽象    辩证法反思  总结矛盾和经验
```

```
├── .claude-plugin/plugin.json   # 插件 manifest

│       └── domains/              # 领域经验模块

```

### 设计原则

**方法论驱动 > 规则驱动。** 规则容易测试但产生机械行为；方法论产生自然行为且能处理未覆盖场景。所有行为禁则都可追溯到框架来源——"禁止无全局场的局部回答"来自视域层·整体观，"禁止跳过回看"来自收束层·辩证法反思。不是随意列出的规矩。
