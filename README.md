# 师兄 (Senior Brother)

> **认知示范 > 答案输出。** 答案只能解决一个问题，认知过程能解决一类问题。

**师兄**不是角色扮演，是一套**可执行的认知方法论**。它让 Claude 展示自己怎么从"不懂"走到"懂"——包括走错的路、卡住的点、排除过的假设、选择 A 放弃 B 的考量。你获得的不是正确答案，而是**下次自己也能找到答案的能力**。

```bash
# 一键安装
git clone https://github.com/AuroBreeze/senior-brother ~/Code/skills/senior-brother && \
  claude plugin install ~/Code/skills/senior-brother
```

`claude plugin install` 会自动注册 skill，之后任何技术对话师兄自动激活。也可手动 `/senior-brother`。

如果已经 clone 过了，直接把这句话发给 Claude Code，它会帮你配置好：

## 为什么是现在

知识的获取成本正在趋近于零。**"知道什么"不再是竞争力。**

但大多数人用 AI 的方式是"提问 → 拿答案 → 复制粘贴"。这产生了一个悖论：解决的问题越多，**自己解决问题的能力越退化**。

更深一层：当 AI 能回答所有已知问题，真正的价值转移到了**"提出正确问题"和"从零到一探索未知"**——而这两种能力，恰恰在"直接拿答案"的使用模式中被闲置了。

LLM 默认是一个"答案输出者"——给你看似合理的回答，但你不清楚哪些是事实、哪些是猜测、推理在哪一步可以质疑。答案是对的，但**你不知道它是怎么来的**。

每次看到 AI 的回答，你自然会想：它为什么会这么做？它当时是怎么想的？再往下追问一层——**它的思维链是基于什么产生的？** 哪些是观察到的事实，哪些是推断，哪些是隐含假设？这一层决定了上面两层。

师兄做的就是把这层摊开。`[事实]` `[推断]` `[假设待验证]` `[猜测]` 四个标记让你看到每一步推理的认知根基，`>` 警告块标注隐含前提。你不用猜"它为什么会这么说"——草稿纸就在桌上。

师兄反过来：

- 不是给你答案，是让你看答案是怎么被找到的
- 不是替你思考，是让你看到思考可以是什么样子
- 不是节省你的认知，是**训练你的认知**

> 快速发展的时代，如果自身认知不能提速，就会被抛弃。知识可以被搜索，答案可以被生成，但**从混乱中找到结构、从现象中锁定根因、从零到一建立理解**——这些能力只能通过反复观看和实践认知过程来获得。

## 什么时候用

**擅长：**
- Debug：从现象一步一步收束到根因，不是猜谜式修改
- 源码分析：从入口追到最底层，建立系统拓扑感
- 架构决策：追问"为什么选 A 不选 B？B 的代价你考虑过吗？"
- 学习新技术：展示"我是怎么学的"，而非知识点罗列

**不适用：** 非技术对话。

> **简单问题会被方法论压垮吗？** 不会。LLM 的默认倾向是偷懒——能短就不会长。面对"这个函数怎么用"之类的问题，模型仍然会简洁回答。方法论的四动作自带例外条款：纯确认、纯追问、非技术回复自动跳过。**方法论是地板，不是天花板——它只在模型最想偷懒的时候（复杂问题面前）拦住它走捷径。**

## 对照实验：用 vs 不用师兄

同一场技术对话（项目评估 + 文件系统抽象设计 + Bug 审查），分别跑两次。完整记录见 `docs/`：

| | 基线（无师兄） | 方法论（师兄） |
|------|---------|-----------|
| **文件** | [`Session_Eval_2025-05-25.md`](docs/Session_Eval_2025-05-25.md) | [`Session_Eval_2025-05-25_Project_Assessment.md`](docs/Session_Eval_2025-05-25_Project_Assessment.md) |
| **长度** | 633 行 | 943 行 |

核心差异——同一个问题"你觉得当前的项目怎么样？"，回答的结构完全不同：

| 维度 | 基线 | 师兄 |
|------|------|------|
| **开局** | 直接列举项目事实 | 先画 Mermaid 架构图，声明"我们现在站在整体架构评估的视点" |
| **认知状态** | 所有陈述平铺，无区分 | 每条标注 `[事实]`/`[推断]`/`[已知]`/`[未知]`——"竞赛就绪度 `[未知]`" |
| **前提声明** | 隐含在论述中 | `>` 警告块显式标注——"此结论在单核场景下成立" |
| **收尾** | 给结论和建议 | 结论 + `[辩证法反思]` 提取可迁移模式——"耦合的根因不是 `#ifdef`，是缺失的运行时信息" |
| **结构** | 项目概览 → 优点 → 问题 → 总结 | 整体观定位 → 现象学 `[事实]` 清单 → 溯因推理 → 逐层评估 → 风险排序 → 回看 |
| **Bug 分析** | 按严重程度分类，描述现象 | 从**不变量破坏**角度切入，每个 bug 标注被破坏的约束 + 最小验证实验 |
| **图** | 0 个 | 5 个 Mermaid 架构/数据流图 |

> 这不是"师兄比默认好"——默认模式的回答也是正确的。区别在于：你拿到答案之后，知不知道这个答案是怎么来的，能不能自己再走一遍。

## 背后的认知框架

上面的七个步骤不是随意编排的。师兄背后是一套三层认知架构——"怎么看 → 怎么想 → 怎么推进"：

```
视域层（怎么看）              引擎层（怎么想）              收束层（怎么推进）
─────────────────          ─────────────────          ─────────────────
整体观      建立系统拓扑    溯因推理    反推最佳解释    现象学观察  先描述，不急解释
不变量思维  锁定被破坏的约束  第一性原理  退回不可拆的基本约束  还原论拆解  拆成最小可验证单元
时间性推理  为什么现在才出问题 格物致知    直接面对一手材料    实践论验证  最小实验说话
                          类比锚定    用日常直觉解释抽象    辩证法反思  总结矛盾和经验
```

### 15 个方法论工具

每个工具独立一个文件，首次触发时按需加载。加新工具只需新增文件，不影响主 SKILL.md。

| 工具 | 触发场景 |
|------|---------|
| 整体观 | 对话开始时；涉及 3+ 组件 |
| 现象学观察 | Debug 开始时；收集事实 |
| 溯因推理 | "这个现象很奇怪" |
| 还原论拆解 | 问题涉及多系统 |
| 第一性原理 | "为什么需要 X？" |
| 收束思想 | 多方向排查需系统排除 |
| 不变量思维 | 锁定根因；哪条约束被破坏 |
| 实践论 | 分析完后设计验证实验 |
| 辩证法反思 | 阶段性收束/回看 |
| 奥卡姆剃刀 | 多假设竞争 |
| 时间性推理 | "为什么 X 正常 Y 不正常" |
| 类比锚定 | "这个概念很难理解" |
| 结构主义 | 分析模块架构/组件关系 |
| 历史方法 | 理解复杂演化系统 |
| 否定之否定 | 二元选择困境 |

### 独立技能扩展

除 15 个方法论工具外，以下技能作为独立模块存在，师兄在合适时机会主动提议调用：

| 技能 | 说明 |
|------|------|
| `senior-brother-start` | 新人引导：3 分钟入门体验（用户问"怎么用""从哪开始"时触发） |
| `senior-brother-index` | 项目认知索引：用方法论建项目认知地图，Init/Update/Query 三种模式 |
| `senior-brother-context-state` | 收束状态快照：把长对话压缩成可继续执行的状态（对话过长或完成验证时触发） |
| `senior-brother-write` | 将技术对话整理成文章（`/senior-brother-write`） |
| `senior-brother-feynman` | 收束复盘：重建诊断路径、标出分叉点、提取可迁移模式、换场景验证（会话收束时触发） |

```
│   │   └── extensions/

└── docs/                            # 设计文档存档
```

### 设计原则

**方法论驱动 > 规则驱动。** 规则容易测试但产生机械行为；方法论产生自然行为且能处理未覆盖场景。所有行为禁则都可追溯到框架来源——"禁止无全局场的局部回答"来自视域层·整体观，"禁止跳过回看"来自收束层·辩证法反思。不是随意列出的规矩。
