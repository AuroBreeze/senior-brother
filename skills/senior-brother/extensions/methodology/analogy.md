# 类比锚定

## 这是什么

不是"打个比方让解释更生动"。类比锚定是一个精确的认知操作：**在对方已有的知识结构中找到一个和当前概念同构的部分，用已知结构的直觉来"降落"未知概念——然后立刻标注类比的失效边界。** 类比的价值不在于它有多像，在于它让你知道什么时候不该再用这个类比。

> 没有标注边界的类比是危险的——它让读者以为自己理解了，实际上只是借了一个熟悉的词。标注边界才是类比完成的标准。

## 三步操作流程

### Step 1: 找同构——不是找相似

同构 = 两者有相同的关系结构，不是表面相似。表和钟表表面相似（都有指针），但它们的内部机制不同构。找的是关系结构的对应。

```
[类比·Step1]
被解释的概念：______（抽象概念）
它的核心关系结构：A 和 B 之间是 ______ 关系，B 和 C 之间是 ______ 关系。
日常直觉中的对应：______（日常场景）中也有同样的关系结构：
  X 和 Y 之间是 ______ 关系，Y 和 Z 之间是 ______ 关系。
结构对应：A↔X, B↔Y, C↔Z，关系统一。
```

### Step 2: 用类比解释——让日常直觉做功

用日常场景的直觉来让人"感觉到"这个概念为什么是自然的。

```
[类比·Step2]
"这就像 ______（日常场景）。在那种情况下，你自然地会 ______（行为/期望），
因为 ______。这个概念本质上是在做一样的事——只不过对象从 ______ 换成了 ______。"
```

### Step 3: 标失效边界——这是必须的

任何类比在某个边界外都会失效。你必须精确说出两条边界：（1）在哪方面不成立？（2）继续用这个类比你会在什么地方做出错误推断？

```
[类比·Step3]
这个类比在以下方面不成立：
1. ______（概念有的但日常场景没有的）
2. ______（日常场景有的但概念没有的）
继续用这个类比，你可能会错误地认为 ______——但实际并非如此。
```

## 四种假执行模式

### 假模式 1：表面相似当结构同构

```
❌ "[类比] Git 分支就像文件夹——你可以把文件放在不同分支里。"
   → 文件夹的物理隔离和 Git 的指针机制完全不同构。
   用文件夹理解分支会导致"分支应该占磁盘空间"的错误直觉。

✅ "[类比] Git 分支就像书签——只是指向某个 commit 的指针，
   不是内容的容器。切换分支 = 移动书签到另一页。
   边界：书签不会记录你折过的角（工作区修改不会自动跟随分支切换）。"
```

### 假模式 2：没标边界

```
❌ "[类比] 虚拟内存就像停车场。"
   → 没说边界。读者可能会认为虚拟内存也有"车位大小统一"的特性——
   但实际上虚拟内存的页面大小和物理页框大小才统一，虚拟地址空间本身不限制。

✅ 每次类比后必须跟一句"但这个类比在 ______ 方面不成立"。
   没有这半句的类比是半成品。
```

### 假模式 3：用一个抽象概念解释另一个抽象概念

```
❌ "[类比] Rust 的 borrow checker 就像类型检查器。"
   → 用同样抽象的概念解释，没有降低认知门槛。

✅ "[类比] borrow checker 就像图书馆的借阅系统——
   同一本书在任一时刻只能有一个借阅者，还了才能借给别人。
   但可以有多个阅览者同时在馆内看（&T 共享引用）。
   边界：图书馆的"到期还书"在 Rust 里没有对应——借用到作用域结束自动归还。"
```

### 假模式 4：类比太长，核心结构被淹没

```
❌ 花三段话讲日常场景的细节，最后说"同理，这就是 X"。
   → 细节淹没了结构对应。

✅ 类比的核心只有一句话。细节只是为了帮助建立结构对应的直觉。
   如果结构对应不够清晰——类比选错了。
```

## 不同领域的示例

### 示例 1: 数据库事务隔离

```
[类比·Step1]
核心关系结构：事务 A 写入数据 → 事务 B 读取数据。
问题是 B 能看到 A 未提交的修改吗？（脏读）A 提交后 B 能看到吗？（可重复读）
日常对应：Google Docs 的协作编辑——你打字的时候，另一个人能看到你打的字吗？

[类比·Step2]
Read Uncommitted = 你在打字，另一个人实时看到你每打一个字（包括你删掉的错字）。
Read Committed = 只有你点了"保存"（commit），别人才看到你的修改。
Repeatable Read = 每个人打开文档时拍了一张快照——你在你这张快照上工作，
   别人保存的内容不影响你已经打开的快照。
Serializable = 一次只能有一个人在文档上工作，排队。

[类比·Step3]
边界：
1. Google Docs 是字符级别的协作，数据库是行/表级别的——粒度不同
2. Google Docs 没有"死锁"的概念——两个人同时编辑同一行会自动合并，
   数据库的 Serializable 可能因为锁冲突而失败
3. 继续用这个类比，你可能会认为 Serializable 是最慢的隔离级别——
   但在某些实现中（如 MVCC + SSI），Serializable 的性能接近 Repeatable Read
```

### 示例 2: 网络协议栈

```
[类比·Step1]
TCP 的重传机制：发送方发包 → 启动定时器 → 超时没收到 ACK → 重传。
日常对应：你让朋友帮你带东西，说了"到了给我发消息"。你等了 30 分钟没消息就打电话确认。

[类比·Step2]
TCP 就像一个谨慎的人——
发送每个包裹（segment）后开始数秒。如果时间到了还没收到"收到了"（ACK），
就重新发一份。对方回复"收到"后，这人才放心地发下一个包裹。

[类比·Step3]
边界：
1. 日常场景中你不会把一个大文件拆成 1460 字节的片段——TCP 的 segment 概念没有对应
2. TCP 的累计 ACK（"收到了第 5 个，前 4 个不用单独确认"）在日常中不自然
3. TCP 的滑动窗口（一次发多个而不用等每个 ACK）用"等消息"类比会丢失——
   事实上发送方不会干等，会继续发后续 segment
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 类比的核心是关系结构对应还是表面相似？ | 能找到 A→X, B→Y 的结构映射 | 只是"都涉及 X 和 Y"的松散关联 |
| 我是否标注了失效边界？ | 至少 2 条具体的失效条件 | 只有类比没有边界 |
| 日常场景是否比原概念更直观？ | 日常场景不需要额外解释 | 用了一个同样抽象的概念来解释 |

## 转换信号

### 何时进入
- 用户说"这个概念很难理解"或"我看了几遍还是不懂"
- 解释一个抽象/多层抽象的概念
- 涉及多个关联概念需要一次性建立直觉

### 何时退出
- Step3 边界标注完毕 → 类比完成。如果概念需要进一步深入，转 [第一性原理] 从约束重新推导
- 发现找不到好的结构同构 → 放弃类比，转 [还原论] 拆解概念或 [格物致知] 从源码学习
- 用户说"我理解了" → 不需要进一步类比的解释了

## 配合使用的工具

| 在前 | 本工具 | 在后 |
|------|--------|------|
| （概念需要解释时直接进入） | **类比锚定** | [第一性原理] 如果需要从约束验证这个概念的正确性 |
| | | [格物致知] 如果需要通过源码/实际运行来确认类比是否正确 |

## 标注前自检清单

- [ ] 我的类比的对应关系是结构同构而非表面相似？
- [ ] 日常场景比原概念明显更简单、更直观？
- [ ] 是否标注了至少 2 条失效边界（具体到"继续用这个类比会错误地认为 X"）？
- [ ] 类比的核心是否能用一句话说清（不需要三段场景描述）？

**未标注边界的类比 = 制造了一个新的误解。**
