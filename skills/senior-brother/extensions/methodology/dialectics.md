# 辩证法反思

## 这是什么

不是"总结一下学到了什么"。辩证法反思是识别一个具体结构：**正题（最初相信什么）→ 反题（发现了什么矛盾）→ 合题（矛盾如何推动理解升级——吸收了双方的合理部分，去掉了错误部分）。** 它的价值不在"总结这次经验"，而在"把这次具体的 bug/设计决策中的矛盾结构抽象成可迁移的思维模式"。

> 关键区分：总结 = "我们做了什么，学到了什么"。辩证法反思 = "这次经历中的矛盾是什么？矛盾如何推动理解？下一次遇到类似矛盾，我该更快地识别出什么？"

## 四步操作流程

### Step 1: 识别矛盾结构

```
[辩证法·Step1]
正题（最初信念/假设）：我以为 ______。
反题（矛盾的发现）：但 ______ 现象和我的预期矛盾。
矛盾的性质：这不是一个简单的"我搞错了"——它暴露了 ______（更深层的问题/假设/设计缺陷）。
```

### Step 2: 跟踪矛盾的推动

矛盾不是障碍——是信息的载体。矛盾推动了你从"不知道"到"知道"的整个运动。

```
[辩证法·Step2]
矛盾如何推动了排查：
1. 矛盾出现时，它迫使我 ______（做了什么调整？改变了什么问题？）
2. 如果当时没有这个矛盾，我可能会 ______（继续走在错误的路上）
3. 这个矛盾本质上是在告诉我 ______（一个我不愿意看到的事实/一个被忽略的假设）
```

### Step 3: 合成 —— 吸收双方的合理部分

```
[辩证法·Step3]
合题（升级后的理解）：
- 从正题中保留了：______（最初信念中正确的部分是什么？）
- 从反题中吸收了：______（矛盾揭示的真相是什么？）
- 去掉了：______（什么样的简化/误解被修正了？）
新的理解比正题和反题都更高一层：______。
```

### Step 4: 沉淀为可迁移的模式

```
[辩证法·Step4]
可迁移模式：
下次遇到 ______（类似特征的情况），我应该更快地 ______（做什么动作），
因为这种模式通常意味着 ______（什么矛盾结构在起作用）。
```

## 四种假执行模式

### 假模式 1：只说"学到了"不说"矛盾是什么"

```
❌ "[辩证法] 这次排查让我学到了 buffer cache 的重要性。"
   → 不是辩证法。这是换了个说法的总结。

✅ "[辩证法] 正题：我以为 cache 的正确性是自明的——把数据缓存起来，
   用的时候返回就行。反题：但当你开始复用缓存项时，
   缓存元数据（dev, blkno）和缓存内容（data）的一致性不再是自明的——
   你必须在每一个修改元数据的点上显式维护这个一致性。
   矛盾的本质：'复用' = 一个对象有了两个身份（旧的和新的），
   你不主动声明身份切换，旧身份的状态就会污染新身份。"
```

### 假模式 2：矛盾结构不对——正题和反题不是互补关系

```
❌ "正题：我以为 bug 在 exec。反题：原来 bug 在 block cache。"
   → 这不是矛盾，这是"我猜错了"。不构成辩证结构。

✅ 正题和反题必须是一个矛盾的两个方面——它们互相否定但各自含有合理成分。
   "正题：软件应该正常工作（依赖 cache 正确）。反题：依赖本身可能出错。
   合题：依赖不是免费的——你必须验证依赖的正确性，
   至少要确保依赖的失败模式是可见的（加 assert）。"
```

### 假模式 3：合题只是"各打五十大板"

```
❌ "合题：有时候要相信直觉，有时候要相信证据。"
   → 这不是合题，这是骑墙。

✅ 合题必须在更高层面统一矛盾双方。"不是'信直觉还是信证据'——
   是'先用直觉生成假设，然后用证据快速淘汰'。
   直觉的合理成分：它能快速指向概率最高的方向。
   证据的合理成分：它是最终的仲裁者。
   合题：直觉生成假设→实验快速淘汰→保留被证据支持的。
   这不是各打五十大板，这是给双方分配了不同阶段的任务。"
```

### 假模式 4：沉淀的模式太具体——只适用于这个 bug

```
❌ "[辩证法] 下次遇到 buffer cache 的问题，先检查 bget 的复用路径。"
   → 这条经验只能用于 bget。下次遇到的是 inode cache 的问题就失效了。

✅ 提取底层模式："下次遇到任何'切换实现/存储/配置后行为异常'的问题，
   优先检查'复用'逻辑——对象从旧的被回收后拿来干新的事情时，
   旧状态有没有被清干净。缓存、连接池、线程池、对象池——都遵循同一个模式。"
```

## 不同领域的示例

### 示例 1: OS — Buffer Cache Debug

```
[辩证法·Step1]
正题：我以为 buffer cache 是简单的——找到 buffer 返回数据。
反题：缓存返回了错误数据。矛盾不在于"缓存有 bug"，
而在于"设计缓存的人（包括我）认为缓存的正确性是自动的"——它不是。

[辩证法·Step2]
矛盾推动排查的方式：epc 处是 addi 却 page fault——这个矛盾迫使我放弃了
"exec 参数有问题""栈有问题"这些上层假设，直接 probe 用户内存。
如果当时没有这个矛盾，我会沿着"修复 exec 参数"的错误方向继续改代码。

[辩证法·Step3]
合题：
- 保留：缓存确实使得系统更高效——这是正确的
- 吸收：缓存正确性不是免费的——每次修改缓存的元数据（dev, blkno），
  必须同步维护内容相关的标志（valid）
- 去掉："cache 会自动工作"的隐含假设
新理解：缓存 = 性能加速 + 一致性负担。负担不会消失，不主动管理就会变成 bug。

[辩证法·Step4]
可迁移模式：任何带有"复用"语义的设计（缓存、连接池、对象池、线程池），
在复用发生时要问——旧状态有没有被清干净？旧身份有没有残留？
这条不变量应该被显式写下来。
```

### 示例 2: 前端 — 从 Class Component 到 Hooks

```
[辩证法·Step1]
正题：Class 组件将状态和方法组织在同一个对象中——逻辑内聚，直觉自然。
反题：Class 组件的生命周期方法（componentDidMount, componentDidUpdate,
componentWillUnmount）把相关的逻辑分散在不同的生命周期中——
同一个数据源的订阅逻辑被拆到了三个方法里。
矛盾："对象的组织结构是内聚的，但逻辑的实际执行路径是离散的。"

[辩证法·Step2]
这个矛盾推动了 React 团队的设计：不按生命周期组织，按"效应"组织。

[辩证法·Step3]
合题：Hooks——
- 保留：状态和组件关联（和 Class 一样）
- 吸收：逻辑按关注点组织而非按生命周期分散（解决了反题的痛点）
- 去掉：this 的绑定问题、HOC 的包装地狱
新理解：不是"Class vs Function"的二元选择——是按"对象的生命周期"还是按"数据的依赖关系"来组织代码。

[辩证法·Step4]
可迁移模式：当你在框架中感到"做一件事要改 5 个不同地方"时，
框架可能是按"系统的运行阶段"而非"开发者的关注点"来组织的。
这不是 bug——是组织维度的选择。Hooks 换了一个组织维度。
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 正题和反题是否构成了真正的矛盾（互相否定但各含合理成分）？ | 同构的矛盾关系 | "我以为 A，其实是 B"——不构成矛盾 |
| 合题是否吸收了双方的合理部分而非各打五十大板？ | 可以说出正题中什么是对的、反题中什么是对的 | "要平衡两者"——没有实质综合 |
| 沉淀的模式是否可迁移（不应该只在当前场景适用）？ | 可以描述为"当 X 时，优先做 Y"的通用模式 | 沉淀太具体——只适用于当前 bug |

## 转换信号

### 何时进入
- 问题解决后 / 阶段性收束 → 回看自然触发辩证法
- 排查过程中出现了明显的"预期 vs 现实"的碰撞 → 标记这个矛盾
- 用户在两种方案之间摇摆 → 辩证法分析矛盾双方

### 何时退出
- Step4 模式沉淀完成 → 回归实践
- 发现矛盾结构不够清晰 → 回到 [现象学] 重新收集事实
- 矛盾涉及未解决的技术问题 → 转 [溯因] 继续排查

## 标注前自检清单

- [ ] 正题和反题是否构成了实质矛盾（不只是"我猜错了"）？
- [ ] 矛盾是否推动了排查/理解（不只是被动的"发现了问题"）？
- [ ] 合题是否吸收了双方的合理部分？
- [ ] 沉淀的模式是否可以迁移到其他场景？
- [ ] 如果三个月后回头看，这次反思的核心洞察是什么？（一句话说得清吗？）

**辩证法反思不是为了显得深刻——是为了让每次经验变成可复用的思维结构。**
