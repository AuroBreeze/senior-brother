# 时间性推理

## 这是什么

不是"什么时候开始出问题的"。时间性推理追问的是：**同样的代码，为什么在某些条件下正常，在某些条件下异常？条件的变化揭示了什么被违反的隐含假设？** 答案往往不在代码本身——代码只是一个函数，它的行为取决于输入。输入变了，隐含假设被暴露，bug 才浮现。

> 关键区分："之前没问题现在有问题"不是时间性推理的终点，是起点。终点是"条件变化揭示了什么设计假设？这个假设在什么条件下会被打破？"

## 四步操作流程

### Step 1: 定时间线 —— 精确到"条件变化"而非"时间变化"

不说"昨天还好好的今天坏了"。说"在配置 A 下正常，切换到配置 B 后异常"或"在输入规模 N 下正常，超过 M 后异常"。

```
[时间性·Step1]
条件 A（正常）：
  - 输入：______
  - 配置：______
  - 环境：______
  - 行为：______
条件 B（异常）：
  - 输入：______
  - 配置：______
  - 环境：______
  - 行为：______
变化了什么？______（具体差异，不能是"系统状态变了"）
```

### Step 2: 找被暴露的隐含假设

每个"在条件 A 下正常"背后都有一条未说出的假设。找出这条假设——它很可能就是根因所违反的。

```
[时间性·Step2]
在条件 A（正常）下，代码隐式假设了 ______。
在条件 B（异常）下，这个假设不再成立，因为 ______。
```

> 这一步是时间性推理的核心。不是"环境变了所以 bug 出现了"——要追问到"代码里写了什么假设，这个假设在环境变化后被破坏"。

### Step 3: 推演 —— 这个假设在什么条件下还会被打破？

不只看当前触发条件。推演其他可能暴露同一假设的条件变化。

```
[时间性·Step3]
同一个隐含假设还可能被以下条件打破：
1. ______（当前还没有触发，但理论上可能）
2. ______
3. ______
```

### Step 4: 泛化 —— 能提炼成规则吗？

```
[时间性·Step4]
可提炼的规则：当 ______（输入/配置/环境特征）发生变化时，
应优先检查 ______（哪类不变量/假设）是否仍然成立。
因为 ______。
```

## 四种假执行模式

### 假模式 1：停止了在"什么时候开始"

```
❌ "[时间性推理] 这个问题从上周五的部署后就出现了。"
   → 只说了时间点，没分析条件变化。

✅ "上周五的部署切换了文件系统（EasyFS→EXT4）。
   条件变化：文件系统的块数从 ~20 增加到 ~339。
   这暴露了 buffer cache 的隐式假设——'buffer 复用不会频繁发生'。
   原来的假设在低压力下成立，在高压力下被打破。"
```

### 假模式 2：把"之前运气好"当成了推理

```
❌ "[时间性推理] 之前没出问题是因为没有并发访问。"
   → "没有并发访问"不是你推导出来的，是你猜的。

✅ 必须提供"为什么之前没有并发"的证据。
   "EasyFS 阶段，所有文件操作都是同步的——代码里 read 之后直接 close，
   没有同时打开两个文件的路径。切换到 EXT4 后，busybox 初始化过程中
   会同时 open 多个 musl 库文件——这在 EasyFS 阶段没发生过。"
```

### 假模式 3：条件变化找到了就停，不追问"暴露了什么"

```
❌ "[时间性推理] 问题在 EXT4 下出现是因为块数多了。"
   → 找到了条件变化，但没回答"为什么块数多会导致问题"。

✅ 必须走到 Step2——"块数多了 → 频繁复用 buffer → bget 的复用路径被触发
   → 暴露了 valid 标志未被清零的 bug"。条件变化不是根因，它暴露的隐含假设才是。
```

### 假模式 4：不推演其他可能的触发条件

```
❌ "只有 EXT4 会触发，EasyFS 不会。"
   → 你的排查结束了，但你的理解没有。同一个不变量被破坏的条件可能不止一种。

✅ Step3："如果在 EasyFS 上创建一个 500MB 的大文件然后用 seek 随机读，
   即使没有 EXT4，buffer 复用也会频繁发生——同样会触发 bug。
   条件不限于文件系统类型，而是'cache 压力超过缓存容量'。"
```

## 不同领域的示例

### 示例 1: 后端 — 服务偶尔 502

```
[时间性·Step1]
条件 A（正常）：QPS < 1000，p99 < 200ms，所有请求正常
条件 B（异常）：QPS > 1500 时，每 100 个请求有 3-5 个 502
变化：QPS 从 1000 涨到 1500

[时间性·Step2]
隐式假设：上游的 timeout 时间（3s）大于下游的最长响应时间。
在条件 A 下这个假设成立（p99=200ms << 3s）。
在条件 B 下，连接池耗尽导致请求排队，排队时间 + 处理时间 > 3s，
上游超时断开，返回 502。

[时间性·Step3]
同一假设还可能被打破：
1. 下游服务出现 GC pause（1-2s），即使 QPS 不变
2. 网络延迟增加（跨 AZ 调用），即使下游处理时间不变
3. 连接池大小被人调小了

[时间性·Step4]
可提炼规则：当 timeout 相关错误随负载上升而出现时，
优先检查"timeout 设置是否考虑了排队时间而不仅仅是处理时间"。
```

### 示例 2: 前端 — 生产环境样式错乱，开发环境正常

```
[时间性·Step1]
条件 A（开发环境）：样式正常
条件 B（生产环境）：样式错乱，部分组件样式丢失
变化：生产环境启用了 CSS 压缩和代码分割

[时间性·Step2]
隐式假设：CSS 的引入顺序和生产环境的合并策略一致。
开发环境中每个组件的 CSS 作为独立文件按组件加载顺序引入。
生产环境将 CSS 合并为一个 bundle——合并顺序和加载顺序不同，
后面的全局样式覆盖了前面的组件样式。

[时间性·Step3]
同一假设还可能被打破：
1. 使用 CSS-in-JS 后，运行时注入顺序和编译时预期不一致
2. 懒加载的组件加载时机不确定，CSS 注入时机也不确定

[时间性·Step4]
可提炼规则：样式问题只在生产环境出现时，
优先检查 CSS 的合并/压缩/排序策略，不是样式本身。
```

## 深度自检

| 自测问题 | 及格线 | 不及格特征 |
|---------|-------|-----------|
| 我是否说清了条件 A 和条件 B 的具体差异（不是"以前正常现在坏了"）？ | 两个条件都有具体参数/配置/输入值 | 条件描述是"在 A 下正常，在 B 下不正常"没有具体差异 |
| 我是否找到了被暴露的隐含假设？ | 一句话说出"代码里假设了什么" | 停在"条件变了所以 bug 出现了" |
| 我是否推演了其他可能触发同一假设的条件？ | 至少 2 个其他场景 | 认为只有当前条件能触发 |

## 转换信号

### 何时进入
- 一个 bug 在特定条件下出现、其他条件下不出现
- "之前没问题，改了 X 之后出问题"
- 问题解决后做回看时——时间性推理是最好的回看工具
- 用户说"为什么在我机器上是好的，在你那是坏的？"

### 何时退出
- Step4 规则提炼完成 → 转 [辩证法] 回看或 [实践论] 验证修复
- 发现条件差异不够具体 → 回到 [现象学] 收集更多条件相关的 [事实]
- 发现隐含假设 → 如果不确定假设是否成立，转 [溯因] 形成假设并验证

## 标注前自检清单

- [ ] 条件 A 和条件 B 的差异是否具体到了参数/配置/输入值层面？
- [ ] 我是否找出了一条"代码里隐含假设了什么"？
- [ ] 这条假设在条件 A 下为什么成立？
- [ ] 这条假设在条件 B 下为什么不成立？
- [ ] 我是否推演了至少 2 个其他可能触发同一假设的场景？

**"之前没问题现在有问题"不是时间性推理。条件变化暴露了什么隐含假设——才是。**
