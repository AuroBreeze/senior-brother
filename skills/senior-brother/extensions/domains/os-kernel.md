# OS Kernel Debugging

## 触发条件

- 用户在内核开发中遇到 bug、异常、panic
- 涉及 page fault、特权级陷阱、内存映射异常
- QEMU 模拟器下的运行时调试
- 分析崩溃现场的寄存器、日志、反汇编

---

## 内核 Debug 的特殊性

`[事实]` 内核调试和用户态调试有本质区别：

- 没有完整的调试器支持（gdb 可用但受限——无法单步穿越特权级切换）
- 崩溃时没有 core dump，只有寄存器快照和有限日志
- 错误可能在本不该出错的地方暴露，根因在更早的步骤
- 不同优化等级（`-O0` / `-O2`）可能表现不同——不是因为优化有 bug，而是优化改变了内存/寄存器的使用模式，暴露了潜伏的问题

---

## 第一步：现象学观察（必须做，不能跳）

`[现象学]` 收到崩溃报告后，第一件事不是解释，是收集事实：

```
必须收集的 [事实]：
- epc / pc：崩溃在哪条指令？
- cause / scause：什么类型的异常？（load fault / store fault / illegal instruction / ecall）
- tval / stval：访问了哪个地址？
- sp：栈指针指向哪里？（高地址还是低地址？）
- ra：返回地址是什么？
- 当前特权级：M-mode / S-mode / U-mode？
- MMU 是否启用？satp 写入的是什么值？
```

`[推断]` 把这些 [事实] 列完后，先别解释。至少确认一个问题：**epc 指向的指令类型和 cause 是否一致？** 如果不一致——比如 `addi`（纯寄存器操作）触发 load page fault（cause=13）——这是一个极其重要的信号。

---

## 第二步：矛盾分析

`[溯因]` 当 epc 指令类型和 cause 矛盾时，最可能的解释：

1. **用户内存内容 ≠ 磁盘文件内容**（代码加载错误）
2. **页表映射到了错误的物理页**（PTE 指向了错误地址）
3. **epc 本身被错误保存**（trap 保存 epc 的代码有 bug）

这三种可以逐步排除。最快的是直接 probe 用户内存：

```
walk_addr(pagetable, epc) → 物理地址 → PA2VA → 直接读机器码
objdump -d binary | grep 机器码
```

如果机器码匹配磁盘文件 → 问题在页表映射。如果不匹配 → 问题在加载路径。

---

## 第三步：逐层排除

`[还原论]` 内核 bug 涉及多个层时，不要同时查所有层。按调用链从最底层往上验证：

```
磁盘读取 (bread / virtio) → block cache (bget) → 文件系统读 (readi / ext4_read) → 加载器 (loadseg / exec)
```

验证策略：从最底层开始，确认每一层的输出是否正确。如果底层输出错了，上层必然错——先锁定底层。如果底层正确，问题在上层，继续往上追。

`[不变量]` 每一层都有自己必须维持的不变量。如果现象是"数据看起来像另一段数据"（而不是随机乱码），优先怀疑 cache/buffer 层的不变量被破坏——某个 buffer 的标签（块号/设备号）和内容（数据）不一致。

---

## 第四步：最小验证实验

`[实践论]` 设计验证实验的原则：

- **最小 probe**：不要在怀疑的代码附近加一堆 printk——加一个精确的检查点
- **对比验证**：同样的操作，用不同的工具（readelf / debugfs / objdump）分别验证，看是否一致
- **二进制对比**：不信任文本输出。读机器码，用 objdump 反查来源

有用的内核 probe 技术：

| 技术 | 用途 |
|------|------|
| walk_addr + PA2VA 读内容 | 验证用户页表中的实际内容 |
| debugfs stat / dump | 验证磁盘上的真实数据和 extent 分布 |
| objdump -d | rg 机器码 | 反查一段错误指令来自哪个文件偏移 |
| readelf -h -l | 验证 ELF 头的 entry point 和 LOAD 段 |

---

## 第五步：锁定后的回看

`[辩证法]` 每次 debug 结束后追问：

- `[不变量]` 哪条不变量被破坏了？代码中是否有显式检查？（如果没有——是否应该加一个 assert？）
- `[时间性推理]` 为什么之前没暴露？什么条件变了？这个条件变化揭示了什么隐含依赖？
- 这次排查的路径可以泛化吗？下次遇到"切换 X 后行为异常"应该优先查什么？

### 通用排查路径总结

```
崩溃 → 收集 [事实]（epc/cause/tval/sp/ra/特权级/MMU状态）
     → 检查 epc 指令类型和 cause 是否一致
         → 不一致 → probe 用户内存，objdump 反查
         → 一致 → 按调用链从底层往上逐层验证
     → 锁定层后追查该层的不变量
     → 回看：为什么之前没暴露？什么条件变了？
```

## 常见陷阱

> ⚠️ **优化等级 trap**：`-O0` 崩溃 `-O2` 不崩溃（或反过来）——根因不是编译器，是优化改变了栈/寄存器使用模式。`-O0` 用栈更多，如果 SP 配置错误会立即暴露。`-O2` 变量放寄存器，碰巧不触发。检查 SP 指向。

> ⚠️ **M-mode MMU trap**：M-mode 下 MMU 默认禁用。如果 SP 指向了虚拟高地址，M-mode trap handler 的第一条栈操作就会写入不存在的物理地址。解决方案：为 M-mode 分配独立的、位于物理内存范围内的栈空间。

> ⚠️ **cache 不变量 trap**：换文件系统后行为异常，且错误内容"看起来有结构"（不是随机垃圾）→ 优先怀疑 block cache 的重绑定逻辑。检查 buffer 复用路径是否同步更新了 valid 标志。

> ⚠️ **特权级委托 trap**：某个中断一直在 M-mode 处理但你以为它被委托给 S-mode 了 → 检查 medeleg/mideleg 寄存器。不同启动路径（裸机 vs 固件）的委托配置可能不同。
