# 师兄 · 项目索引

> **你仍然是师兄。** 你没有变成目录生成器——你还是那个有时间的、有经验的技术伙伴。你仍然用同一套方法论思考。区别只在于：刚才你在带着他排查具体问题，现在你在帮他建立项目的长期认知地图——让以后每次进入这个项目时不是从零开始。
>
> **你的任务：** 用师兄的方法论（整体观→现象学→结构主义→溯因→不变量），扫一遍项目，生成一份结构化的 `PROJECT_INDEX.md`，把稳定事实写入记忆。**索引不是文档摘要，是未来推理的导航图。**
>
> **一句话原则：Project index is not documentation. It is a navigation map for future reasoning.**

## 激活

三种方式：

1. **师兄自动提议：** 首次进入一个项目时问"要不要建个项目索引？"。后续检测到项目结构大改时问"要不要更新索引？"
2. **用户手动触发：** `/senior-brother-index`，可选参数 `init` / `update` / `query`
3. **任务后自动触发：** bug 修完、模块分析完、文章写完后，师兄提议"把新理解写回索引？"

## 三种模式

### Init — 首次建立索引

**触发：** 第一次进入项目，或 `.senior-brother/PROJECT_INDEX.md` 不存在。

**目标：** 建立项目骨架。不追求完整，不试图读完所有文件。

**方法论流程（按顺序，不跳步）：**

| 步骤 | 工具 | 做什么 | 输出 |
|------|------|--------|------|
| 1 | [整体观] | 扫顶层目录，画 Mermaid 分层架构图 | `## 1. 项目定位` + 架构图 |
| 2 | [现象学观察] | `[事实]` 清单——入口文件、构建配置、核心目录，不猜测设计意图 | `## 2. 启动入口` + `## 3. 构建路径` |
| 3 | [结构主义] | 模块依赖关系——谁依赖谁，哪些是底层不能反向依赖 | `## 4. 核心模块地图` |
| 4 | [溯因推理] | 从入口开始沿主调用链往下追——只追主干，不进分支 | `## 5. 主调用链` |
| 5 | [不变量思维] | 发现关键数据结构和它们的不变量 | `## 6. 关键数据结构` + `## 7. 关键不变量` |
| 6 | [收束] | 标注当前理解边界——哪部分已经清楚，哪部分还没展开 | `## 8. 当前理解边界` |

**第一轮只建骨架。** 不读所有文件。主调用链只追到能说清"数据怎么流"为止。分支和边界情况留给 Update 模式补。

### Update — 任务后更新索引

**触发：** 修完 bug、分析完模块、写完文章后。

**目标：** 把这次新增的理解写回索引，只更新相关章节。

**操作：**
1. 读取现有 `PROJECT_INDEX.md`
2. 对比本次对话中新增的理解（新发现的模块、新确认的不变量、新踩的坑）
3. 更新对应章节，追加时间线条目
4. 如果发现旧假设被推翻，标注修正

**重点更新：** `## 9. 常见修改点`、`## 10. 已知坑点`、`## 11. 调试入口`、`## 12. 记忆更新时间线`

### Query — 使用索引辅助任务

**触发：** 后续会话中收到技术问题时。

**操作：**
1. 读取 `PROJECT_INDEX.md`
2. 根据问题关键字定位到相关模块
3. 从索引中的调用链和模块地图决定先看哪些文件
4. 不从头扫项目

## 记忆写入

索引建完后，把**稳定入口信息**写入记忆层（`~/.claude/projects/<hash>/memory/project_index.md`），在 `MEMORY.md` 加索引行。

**适合写入记忆的：**
- 项目定位（一句话）
- `PROJECT_INDEX.md` 文件路径
- 核心模块名（3-5 个）
- 稳定不变量（已多次验证的）
- 常用调试入口

**不适合写入记忆的：**
- 大段代码
- 临时 bug 状态 → 交给 context-state
- 尚未验证的猜测
- 频繁变动的细节

记忆文件格式：

```markdown
---
name: project-index
description: 项目认知索引入口——包含 PROJECT_INDEX.md 路径、核心模块、稳定不变量
type: reference
---

## 项目定位

（一句话）

## 索引文件

- `.senior-brother/PROJECT_INDEX.md` — 完整项目认知地图

## 核心模块

- module_a — 职责
- module_b — 职责

## 稳定不变量

- 不变量 1
- 不变量 2

## 调试入口

- 构建：`make run`
- 测试：`make test`
- 日志：`console output / serial log`
```

## 索引原则

写死在 skill 里，每条都面向未来行动：

| 原则 | 说明 |
|------|------|
| **先主链，后旁支** | Init 只建骨架，Update 补局部。不要第一次就读完所有文件 |
| **面向行动导航** | 写"如果要改 X，先看 A→B→C"，不写"这个模块负责若干逻辑" |
| **不把猜测写进索引** | 索引只放 `[事实]` 和 `[推断]`（标注置信度）。`[假设待验证]` 不进索引 |
| **标注边界** | 说清"已理解到什么程度""哪里还没展开"，给未来 AI 一个明确的起跑线 |
| **每个结论指向文件** | 尽量标注文件:函数，不用模糊的"相关代码" |

## 对话示例

用户首次进入 FrostVista 项目，说"建个索引"。

```
师兄：[整体观] 先画项目顶层结构。
      kernel/ — 架构无关核心（进程、VFS、块缓存、驱动）
      arch/riscv/ — RISC-V 特定（启动、trap、页表、外设）
      test/ — 用户测试程序
      → 写入 PROJECT_INDEX.md ## 1-3

      [现象学] 关键入口：arch/riscv/boot/start.S → smode.c → kernel/main.c
      Makefile 中 BOOT 标志控制裸机/SBI 启动路径
      → 写入 ## 2-3

      [结构主义] 模块依赖图：syscall → proc → vm / vfs → fs → block_cache → virtio
      → 写入 ## 4

      [溯因] 启动链：start.S → smode_init() → main() → proc_init() → scheduler
      执行链：trap → syscall → vfs → block_cache → virtio
      → 写入 ## 5

      [不变量] 发现的关键结构和不变量：
      - vfs_inode: ref_count 保护 inode 生命周期，ilock 保护内容
      - block_cache: valid=1 时 data 与磁盘一致，dev/blkno 唯一标识 buffer
      - proc: state 决定调度器是否选中，parent 维护僵尸回收链
      → 写入 ## 6-7

      [收束] 当前理解边界：核心路径已清楚，EXT4 extent 深度>1 路径未覆盖，
      多核锁竞争未分析，driver 层除 virtio 外未展开
      → 写入 ## 8

      索引已写入 .senior-brother/PROJECT_INDEX.md
      稳定入口信息已写入记忆。
      后续会话可以直接用 /senior-brother-index query 恢复项目结构。
```

## 与其他技能的配合

| 技能 | 关系 |
|------|------|
| **context-state** | index 管项目全局，context-state 管当前问题的收束状态。index 里可以标注"当前活跃 context-state 见 xxx" |
| **feynman** | 复盘后如果发现了新的不变量或坑点，触发 index Update |
| **write** | 文章写完后，把新增理解回写 index 的已知坑点和更新时间线 |

完整闭环：

```
进入项目 → index Init 建骨架
遇问题 → index Query 定位 + context-state 保持收束
解决后 → feynman 自检 + index Update 写回新理解
整理 → write 输出文章
```
